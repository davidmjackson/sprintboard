# S4.3 — Delete ticket — Design

**Story:** E4 / S4.3. **Date:** 2026-07-16. **Status:** approved, pre-implementation.

## Acceptance criteria (from `docs/sprintboard_phase1_backlog.md`)
- Deleting removes the ticket **after a confirm step**.
- A deleted ticket **disappears from board and backlog**.

## Goal and showcase framing
Delete a single ticket from its detail view, the way Jira does it: an **"Actions" kebab
(`⋯`) menu** in the ticket header opens a dropdown whose items include **Delete**; Delete
opens a confirmation, and confirming removes the ticket. The Jira look is deliberate (this
is a portfolio demo that must read as Jira at a glance), a rebrandable skin over
Sprintboard's own model. Fidelity of the *interaction* — kebab → menu → Delete → confirm —
is the point.

## Scope discipline (locked Phase 1)
This story is **exactly**: a kebab Actions menu holding **one** item (Delete), an
`AlertDialog` confirm, a `deleteTicket` write, and removal from the shared context. It
deletes a **ticket** — not the board (fixed four columns, not an entity) and not a project
(no delete-project story exists in Phase 1). Explicitly **out of scope**:
- Every other Jira Actions-menu item (Clone, Move, Archive, Log work, Export, …) — not built.
- Block / unblock — **S4.4** (its menu items will join this same kebab menu later).
- Bulk delete, delete from board/backlog cards, undo/restore, soft-delete — not built.

The kebab menu is a deliberate **forward seam**: S4.4 adds Block/Unblock as sibling items.
We build only Delete now (YAGNI); the menu component is the reusable part.

## Resolved design decisions
1. **Delete lives in the detail modal, behind a kebab Actions menu.** Not on the board/backlog
   cards (keeps the compact cards clean; the modal is the "open a ticket → act on it" surface).
   Mirrors the Jira screenshot the user supplied.
2. **Confirm = a nested `AlertDialog`** ("Delete {KEY}? This can't be undone." · Cancel /
   Delete-destructive). Radix handles an `AlertDialog` nested inside the detail `Dialog`
   cleanly. The confirm step is AC-required — a bare menu click must never delete.
3. **Delete is NOT optimistic.** Unlike inline field edits (which must feel instant), a delete
   is a single terminal action: `await deleteTicket`, and only on success remove the row and
   close the modal. On failure, surface the modal's existing error banner and keep the ticket.
   Optimistically removing then re-adding a "deleted" ticket on error is jarring and confusing.
4. **No refetch.** Removal is a local context mutation (`onTicketDeleted(id)`), mirroring the
   S4.1 optimistic-append and S4.2 in-place-update patterns and the deliberate avoidance of the
   S4.1 stale-refetch race.

## Verified mechanics (checked against `docs/sprintboard_phase1_schema.sql`)
- **`tickets_owner` is `FOR ALL`** (line 339) — it covers DELETE. A cross-tenant delete is
  filtered by the `USING` clause to **zero rows** (RLS filters, it does not raise — the same
  USING path proven for the S4.2 cross-tenant update). No new policy is needed.
- **Deleting a parent epic is safe.** `tickets_epic_fk` is
  `references tickets (id, project_id) on delete set null (parent_epic_id)` (line 176–177):
  deleting a ticket that other tickets reference as their epic **sets those children's
  `parent_epic_id` to null** — the children survive, they just lose the link. No cascade-delete,
  no RESTRICT block. (No UI sets `parent_epic_id` until S4.5, so this is latent-but-correct now.)
- **Keys are never reused.** Delete does not touch `project_counters`; a deleted `MP-5` leaves a
  gap and the next ticket is still `MP-6`. Correct and intentional (Jira behaves the same) —
  never "reclaim" the number.

## Layout — the Actions menu in the detail modal header
`TicketDetailDialog`'s header currently holds `key · type · status` (read-only). Add a **kebab
(`⋯`) button** at the header's right edge (Jira position), opening a **shadcn `DropdownMenu`**
(radix-ui, already installed — no new dependency). The menu contains one **destructive** item,
**Delete**. Selecting Delete opens the `AlertDialog`. The kebab needs an accessible name
(`aria-label="Ticket actions"`); the Delete item is reachable by role/name in tests.

New shadcn primitives (both from the unified `radix-ui` package, no new deps):
- `src/components/ui/dropdown-menu.tsx`
- `src/components/ui/alert-dialog.tsx`

## Interaction and semantics
1. Click kebab → dropdown opens. Click **Delete** → dropdown closes, `AlertDialog` opens.
2. **Cancel** (or Escape / overlay) → close the confirm, nothing happens, ticket intact.
3. **Delete** (destructive) → button enters a brief pending/disabled state → `await
   deleteTicket(ticket.id)`.
   - **Success** → `onTicketDeleted(ticket.id)` (removes from board + backlog) → close the
     confirm AND the detail modal (`onOpenChange(false)`).
   - **Failure** (`ok: false`, or zero rows returned) → close the confirm, keep the detail
     modal open, show the existing error banner ("Could not delete this ticket. Please try
     again."). Ticket stays.

## Data layer — `src/lib/tickets.ts`
- Add `deleteTicket(id: string): Promise<DeleteTicketResult>`:
  `supabase.from('tickets').delete().eq('id', id).select()`.
  - `.select()` returns the deleted rows, so we distinguish a real delete (one row) from RLS
    filtering to **zero rows** — a zero-row result is treated as `{ ok: false }`, not silent success.
- `DeleteTicketResult = { ok: true } | { ok: false; error: 'unknown' }`, consistent with
  `createTicket`/`updateTicket`. On a Supabase error or an empty result set → `{ ok: false }`.

## State flow — `ProjectShellContext`
- Extend the context with `onTicketDeleted(id: string)` that removes the matching row from
  `loaded.tickets` (guarded by `prev.projectId === project.id`, like `onTicketUpdated`). **No
  refetch.** Board and backlog read from this context, so the row vanishes from both surfaces
  immediately → satisfies "disappears from board and backlog."
- The detail modal is rendered from `ProjectShell` with `key={selected?.id}`; on delete we set
  `selectedId = null`, which both closes and (via the key going to `'none'`) unmounts the modal.

## Testing
- **Unit:**
  - `deleteTicket` — success shape, and zero-rows → `{ ok: false }` (no false success on RLS filter).
  - `TicketDetailDialog` — kebab opens the menu; Delete opens the confirm; **Cancel deletes
    nothing** (no `deleteTicket` call, ticket intact); confirming calls `deleteTicket`, fires
    `onTicketDeleted`, and closes the modal; a failed delete shows the error and keeps the modal.
  - `ProjectShell`/wiring — after `onTicketDeleted`, the ticket is gone from board and backlog.
- **Live integration (`src/test/tickets.integration.test.ts`):**
  - Owner deletes their ticket → row is gone on independent re-read (positive control).
  - **Cross-tenant** delete affects **zero rows**, and the target row is **independently re-read
    and confirmed to still exist** (negative + independent control — RLS filters, does not raise:
    the same discipline as the S4.2 update tests).
- **jsdom note:** radix `DropdownMenu`, like radix `Select`, may need pointer-API mocks
  (`hasPointerCapture`, `scrollIntoView`, `PointerEvent`) to open under `userEvent` in jsdom
  (see [[sprintboard-frontend-conventions]]). Plan Task 1 **spikes** this: if the standard
  component opens cleanly (or with a small, once-only mock added to the test setup), use it; if
  it proves fussy, fall back to a minimal custom menu (a button toggling a `role="menu"` list).
  `AlertDialog` is expected to behave like the existing `Dialog`, which tests fine in jsdom.

## Review plan
- Standard task review (spec + quality) after each task.
- **Deep multi-agent (Ultracode) review is NOT mandatory here:** this diff does not change RLS
  machinery, secrets, or the CI gate — `tickets_owner` already covers DELETE and is proven. It
  is a *new destructive path*, though, so the live cross-tenant delete test is required, and the
  final whole-branch review should include a lens on: the zero-rows-is-not-success handling, the
  modal close/unmount interaction with the S4.2 `key`/mounted guards, and that Cancel is truly
  inert. Escalate to a deep pass only if that review surfaces a concurrency/isolation concern.
