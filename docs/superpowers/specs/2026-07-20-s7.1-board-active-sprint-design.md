# S7.1 — Render the four fixed columns (active-sprint board)

**Story:** S7.1 (Epic E7 Board). **Date:** 2026-07-20.

## Acceptance criteria (from `docs/sprintboard_phase1_backlog.md`)

- The board shows To Do, In Progress, In Review, Done.
- It renders **the active sprint's** tickets in the right columns.
- An empty column renders cleanly.

## The one real decision, resolved

The board today is **sprint-blind**: `BoardTab` renders *all* the project's tickets,
partitioned by status. An existing S5.1 test (`BoardTab.test.tsx:102`) asserts that on
purpose ("the sprint filter is the backlog, not the board").

AC2 says the board renders *the active sprint's* tickets. That is the canonical Scrum
sprint board and contradicts the sprint-blind behaviour. **Resolved (owner decision,
2026-07-20): active-sprint-only.** The board filters to the tickets whose `sprint_id` is
the active sprint's id. Consequences:

- No active sprint → four empty columns (nothing is in flight).
- Future-sprint tickets → hidden (planned, not started).
- Completed-sprint Done tickets → hidden (sprint history, S6.4 — not on the board).
- Backlog tickets (`sprint_id is null`) → hidden (they live in the Backlog tab).

The S5.1 sprint-blind test is deliberately reversed by this story.

## Design

### 1. New pure rule module — `src/lib/board.ts`

```ts
export function selectActiveSprint(sprints: readonly Sprint[]): Sprint | null
```

Returns the sprint with `status === 'active'`, or `null`. The board's rule lives here and
nowhere else, mirroring how `src/lib/backlog.ts` owns the backlog rule (never inlined in a
component). The **one-active-per-project** invariant is enforced by the
`sprints_one_active_per_project` partial unique index, so `.find(...) ?? null` returning the
first match is safe — documented in the function, not re-checked.

The ticket-membership half already exists and is **reused, not duplicated**:
`selectSprintTickets(tickets, sprintId)` in `backlog.ts` (the `sprint_id === sprintId` rule,
strict `===` on a `string` id). `board.ts` therefore adds only the active-sprint selector.

### 2. `BoardTab` composition

```ts
const activeSprint = selectActiveSprint(sprints)
const boardTickets = activeSprint ? selectSprintTickets(tickets, activeSprint.id) : []
// then partition boardTickets by status into the four fixed columns (unchanged)
```

Column rendering (headings from `TICKET_STATUS_LABELS`, `TICKET_STATUSES` order, per-column
"No tickets yet." empty text, `TicketCard` + `onOpenTicket`) is unchanged.

### 3. The board now depends on TWO reads

Before this story the board consulted only `ticketsPhase`. It now also needs `sprints` to
know which sprint is active, so it consults `sprintsPhase` too. If the sprints read failed
or is still loading, the board **must not** confidently render an empty board — that would
resurrect the exact "a distinct state wears the empty state's face" defect the component's
S4.6 docblock already guards against. Order of checks in `BoardTab`:

1. `ticketsPhase === 'failed'` → `<LoadFailure resource="tickets" onRetry={onRetry} />`
2. else `sprintsPhase === 'failed'` → `<LoadFailure resource="sprints" onRetry={onRetry} />`
3. else `ticketsPhase === 'loading' || sprintsPhase === 'loading'` → `Loading…`
4. else (both loaded) → render the board

**Tickets-first failure priority** (step 1 before 2): both reads share one `onRetry` (the
shell reloads both together and both typically fail together — a paused DB takes both), so a
single alert with a single Retry is correct. When both failed, showing the tickets failure
is the natural primary. `LoadFailure`'s `resource` union stays closed at `'tickets' |
'sprints'` — a **deliberate security control** (it owns the copy so a raw PostgREST error
string can never reach `role="alert"`). This story does **not** widen it; it reuses the
existing `sprints` case.

### 4. No-active-sprint caption (judgment call — not in the ACs)

When sprints are loaded but none is active, the board renders the four columns (AC1 stays
literally true) with a one-line caption above the grid:

> No active sprint — start one from the Sprints tab.

Rationale: four "No tickets yet." columns when the truth is "you haven't started a sprint"
is the same anti-pattern the component's S4.6 docblock names — a distinct state wearing the
empty state's face. The caption is the honest signal. It is a one-line removal if unwanted.
**Approved by owner, 2026-07-20.**

### 5. Test rewrite

`src/lib/board.test.ts` (new): `selectActiveSprint` — returns the active sprint; returns
`null` for an all-future/complete list and for `[]`; picks the active one out of a mixed
list.

`BoardTab.test.tsx` (rewritten fixtures): tests that expect tickets on the board now supply
an active sprint in `sprints` and tag tickets with its id. New/changed assertions:

- Active-sprint ticket shows in its status column.
- A backlog ticket (`sprint_id: null`) does **not** show (reverses S5.1 line 102).
- A future/completed-sprint ticket does **not** show.
- No active sprint (sprints loaded, none active) → four empty columns + the caption.
- `sprintsPhase === 'failed'` → the sprints failure alert, not an empty board.
- `sprintsPhase === 'loading'` → `Loading…`, not an empty board.
- Existing failure/loading/empty-state and open-ticket tests preserved (updated fixtures).

## Deliberate omissions (YAGNI)

- No active-sprint **name** chrome on the board — not in the ACs; board chrome can come later.
- No drag-to-change-status — that is S7.2.
- No blocked marker/filter on the board — that is S7.3.

## Out of scope / untouched

Schema, RLS, data layer (`listTickets`/`listSprints`), the shell's read/mutation logic, and
`LoadFailure`'s union all stay exactly as they are. This is a client-render story.
