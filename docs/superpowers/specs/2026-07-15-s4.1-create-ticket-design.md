# S4.1 — Create ticket with atomic key generation

**Epic:** E4 Tickets · **Jira:** SPRIN-22 (confirm at ship) · **Branch:** `s4.1-create-ticket`

## Story

Create a ticket in a project. The ticket receives a `PROJECTKEY-N` key from the
project's atomic counter, and lands on the Board (in its column) and the Backlog.

## Acceptance criteria

- A new ticket receives key `PROJECTKEY-N`, with `N` from the atomic counter.
- Two tickets created in quick succession get consecutive, unique numbers — no gaps
  from count-based logic, no collisions.
- New tickets default to status **To Do**.

## Scope

First ticket **write** path. Create + display only. The create form is "rich"
(summary, type, description, story points, labels, acceptance criteria). Status is
**not** chosen on create — the DB default `'todo'` is AC #3. **No assignee** — under
owner-scoped RLS with no teams the only assignee is the owner, so a picker is
degenerate and, worse, would require the S4.2 `profiles` read-model decision (the
`display_name`-is-email leak). Deliberately deferred.

**Parked (later E4 stories), not built here:** ticket detail/edit modal (S4.2), delete
(S4.3), block/unblock (S4.4), epic context + deliverables + parent-epic (S4.5), and the
real assignee picker.

## Key mechanics (preserve exactly — CLAUDE.md)

- **The key is the database's job.** `assign_ticket_key()` (BEFORE INSERT trigger)
  atomically increments `project_counters.last_number` under a row lock and sets
  `new.number` / `new.key`. The client **never** sends `key` or `number`; the
  `TicketInsert` type (`Omit<…, 'key' | 'number'>`) makes that untypeable. Never
  generate keys with `count(*)`.
- **`tickets` has no `owner_id`.** The `tickets_owner` RLS policy scopes through the
  project join (`exists (select 1 from projects p where p.id = tickets.project_id and
  p.owner_id = auth.uid())`). So `createTicket` inserts `project_id` and content
  fields only — no owner column.
- **Cross-tenant insert is doubly blocked:** the `counters_owner` policy denies the
  counter `update` (0 rows → `number := NULL` → `NOT NULL` abort), and `tickets_owner`
  `with check` denies the insert. Either is the security boundary. The live test proves
  it stays that way.

## Design

### Data layer — `src/lib/tickets.ts`

```
createTicket(input: {
  projectId: string
  summary: string
  type: TicketType
  description?: string
  storyPoints?: number
  labels?: string[]
  acceptanceCriteria?: string
}): Promise<CreateTicketResult>
```

`CreateTicketResult = { ok: true; ticket: Ticket } | { ok: false; error: 'unknown' }`.
Inserts `{ project_id, summary, type, description, story_points, labels,
acceptance_criteria }` — omitting `key`, `number`, `status`, `owner_id`. `.select()
.single()`; any Postgres error maps to `'unknown'` (no user-correctable duplicate case).

```
listTickets(projectId: string): Promise<Ticket[]>
```

`.eq('project_id', projectId).order('number', { ascending: true })`. The project
filter is required: RLS scopes to the owner across *all* their projects.

### Ticket state in `ProjectShell`

`ProjectShellContext` grows to
`{ project: Project; tickets: Ticket[]; loadingTickets: boolean; addTicket: (t: Ticket)
=> void }`. The shell loads tickets once (effect keyed on `project.id`) and provides
them through the nested `<Outlet context>`; both tabs consume the same list. The header
gains a **New ticket** button opening `CreateTicketDialog`, wired optimistic-insert +
background refetch (the S3.2 project pattern): `addTicket` inserts the returned ticket
immediately, then a refetch reconciles ordering.

### Components

- **`CreateTicketDialog.tsx`** — shadcn Dialog + Form + RHF + zod, mirroring
  `CreateProjectDialog`. Fields: summary (required, 1–200), type (Select, default
  `story`), description (Textarea, optional), story points (number Input, optional,
  int ≥ 0), labels (Input, comma-separated → `string[]`), acceptance criteria (Textarea,
  optional). On success: `onCreated(ticket)`, close, reset.
- **`TicketCard.tsx`** — key (mono) + type badge (`TICKET_TYPE_LABELS`) + summary.
  Reused by the Board columns; the Backlog renders a flat ordered list of the same data.
- **`BoardTab` / `BacklogTab`** — read `tickets` from context. Board groups by `status`
  into the four `TICKET_STATUSES` columns (empty column keeps "No tickets yet."); Backlog
  is a flat list ordered by number, empty-state when none.

### `domain.ts`

Add `TICKET_TYPE_LABELS: Record<TicketType, string>` = Epic / Story / Bug / Task. Type
display names belong only here, same rule as `TICKET_STATUS_LABELS`. Covered by a
`domain.test.ts` exhaustiveness assertion.

### New UI primitives

Add shadcn `Select` (pulls `@radix-ui/react-select`) and `Textarea` (dependency-free),
consistent with the existing shadcn dialog/form components.

## Tests (TDD, from the ACs)

**Live integration — `src/test/tickets.integration.test.ts`** (`@vitest-environment
node`, `describe.skipIf(!hasRlsCredentials)`, mirrors `projects.integration.test.ts`):

- Creates a project, then a ticket → `key === "<PROJKEY>-1"`, `number === 1`,
  `status === 'todo'` (AC #1, AC #3).
- Two tickets in succession → `number` values are consecutive and unique (AC #2).
- User B inserting a ticket into A's project is rejected (error set, no row) — the
  cross-tenant security property.
- `afterAll` deletes created tickets then the project.

**Unit:**

- `tickets.test.ts` — `createTicket` maps a Postgres error to `{ ok: false, error:
  'unknown' }` and a success to `{ ok: true, ticket }` (mocked supabase, mirroring
  `projects.test.ts`).
- `CreateTicketDialog.test.tsx` — required-summary validation; submit calls
  `createTicket` with parsed fields; a comma-separated labels string becomes a
  `string[]`; success calls `onCreated`.
- `BoardTab.test.tsx` / `BacklogTab.test.tsx` — render tickets from a provided context:
  Board groups a ticket into its status column; empty columns show the empty state;
  Backlog lists by number.
- `ProjectShell.test.tsx` — the New ticket button is present; tickets from context
  render; the existing guard/loading/deep-link tests still pass with the new context.
- `domain.test.ts` — `TICKET_TYPE_LABELS` has an entry for every `TICKET_TYPES` value.
