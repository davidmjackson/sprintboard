# S2.1 — Email & password signup with auto profile (design)

**Story (E2 Authentication):** a new user signs up with email + password; the app
creates the auth user and a matching `profiles` row; display name defaults to email
when none is given; a duplicate email is rejected with a clear message.

This story also lands the **authentication foundation** for the rest of E2 (router,
auth-state context, guard, shared auth UI). Decisions below were taken with the user;
the remainder are best-practice defaults chosen autonomously.

## Decisions (locked)

1. **Add `react-router-dom` now.** S2.3 (protected routes) and E3 (project routes)
   need it; introducing it here means those stories add coverage, not architecture.
2. **shadcn Form (react-hook-form + zod resolver).** The idiomatic stack for
   shadcn/ui; accessible per-field errors; reused by every later form. Adds
   `react-hook-form` + `@hookform/resolvers`.
3. **Live integration test + component tests.** The trigger ACs (profiles row exists,
   display_name defaults to email) are proven against the real database, in keeping
   with the RLS suite. The frontend ACs (validation, duplicate message) are component
   tests with a mocked client.
4. **New service-role test secret.** Verifying the `profiles` row and deleting the
   throwaway user need service-role. A test-only `SUPABASE_SERVICE_ROLE_KEY`
   (**non-`VITE_`**) is added to the harness, CI, and `.env.local`.

## Scope of this story vs the rest of E2

Introducing a router mid-project unavoidably lands shared scaffolding early. To keep
the slice honest:

- **S2.1 delivers (fully tested):** the signup page and flow, the auth-state context,
  and the trigger/duplicate ACs.
- **S2.1 lands as foundation (its dedicated tests come later):** `<BrowserRouter>`,
  `<AuthProvider>`, `<RequireAuth>`, `<AuthLayout>`, and a **minimal working login
  form** so the guard's redirect target exists and a signed-up user can get back in.
  S2.2 hardens login/logout/session-persistence with its own tests; S2.3 adds the
  redirect assertions. No re-architecting is required for those — they add tests and
  polish to components that already exist.

This is the intended reading of "one small PR per story": the foundation is a
one-time cost paid at the first auth story, not scope creep.

## Signup flow & validation

- **Fields:** `email` (required, `z.string().email()`), `password` (required,
  `z.string().min(8)`), `displayName` (optional). The optional field exists precisely
  so the "defaults to email when none is provided" AC has both paths; left blank by
  default, exercising the default.
- **Submit:** `supabase.auth.signUp({ email, password, options: displayName ? { data: { display_name: displayName } } : undefined })`.
  Omitting `data.display_name` lets the `handle_new_user` trigger fall back to
  `new.email` — the AC's default path.
- **Success:** `onAuthStateChange` fires → context session set. Navigate to `/`.
- **Duplicate email (robust, confirmation-agnostic):** treat **either**
  - an error whose message matches `User already registered`, **or**
  - a returned `data.user` with an empty `identities` array (Supabase's
    anti-enumeration signal when email confirmation is *on*)

  as "email already in use", and show one clear message:
  *"An account with this email already exists. Try logging in instead."* This removes
  any dependency on the project's email-confirmation setting.
- **Other errors:** field errors via `FormMessage`; submit/network errors via a
  form-level alert (`form.setError('root', …)`): a generic
  *"Something went wrong. Please try again."*

## Testing

**Component — `src/routes/SignupPage.test.tsx`** (jsdom, `supabase` mocked):
- invalid email → field error, no `signUp` call.
- password < 8 → field error.
- valid submit → `signUp` called with the exact args; blank displayName ⇒ `options`
  omitted (default path).
- `signUp` rejects `User already registered` → the clear duplicate message renders.
- `signUp` resolves with `user.identities === []` → same duplicate message renders.

**Live integration — `src/test/signup.integration.test.ts`** (real Supabase, admin):
- sign up a fresh randomised email (no displayName); via the **test-only admin
  client**, read `profiles` for the new id → `display_name === email`.
- (second case) sign up with a displayName → `profiles.display_name === displayName`.
- `afterAll`/`afterEach`: `admin.auth.admin.deleteUser(id)` for every user created —
  **no churn**.
- Gated exactly like the RLS suite: skips loudly locally when the key is absent,
  **refuses to skip in CI** (`assertServiceRoleOrExplain`).

Randomised email without `Math.random()` (banned in this project's tooling only for
workflow scripts, but avoided here too for determinism of failures): derive from
`crypto.randomUUID()`.

## Harness / infra changes (security-sensitive — precise wording required)

1. **`vite.config.ts`:** add `'SUPABASE_SERVICE_ROLE'` to the `loadEnv` prefix list so
   the test process (only) sees `SUPABASE_SERVICE_ROLE_KEY`. It is **not** `VITE_`-
   prefixed, so Vite never inlines it into the bundle.
2. **`src/test/supabase-clients.ts`:** add `hasServiceRoleKey`,
   `assertServiceRoleOrExplain()`, and `adminClient()` (a service-role client with
   `persistSession:false`). Lives under `src/test/`; **app code must never import it.**
3. **`.env.example`:** add `SUPABASE_SERVICE_ROLE_KEY=` under a comment that draws the
   distinction sharply: the *browser* slot must never hold this key (unchanged rule);
   this *test-only, non-`VITE_`* slot legitimately does. Resolves the apparent
   contradiction with the existing line-7 warning.
4. **`.github/workflows/verify.yml`:** pass `SUPABASE_SERVICE_ROLE_KEY:
   ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}` into the `npm run verify` step's env,
   alongside the existing RLS secrets. The diff touches the gate job — reviewed with
   the caution the file's own comment demands (no `if:`, no matrix, no paths filter
   added).
5. **`check-bundle.mjs`:** unchanged. The new key is non-`VITE_`, so it cannot enter
   the bundle; check-bundle remains the browser backstop and stays green.
6. **`CLAUDE.md`:** document the test-only admin client and the new CI secret
   (server-side only, never bundled); **update the test-count tripwire** — the current
   51 rises by the new integration + component tests. State the new exact numbers once
   verified.

### Manual steps the user performs (I cannot handle the secret value)

- Put `SUPABASE_SERVICE_ROLE_KEY=<value>` in `.env.local`.
- `gh secret set SUPABASE_SERVICE_ROLE_KEY` for the repo (CI gate).

Until both are done the integration test skips locally and CI cannot pass — so these
are the final gating step before merge, walked one at a time.
