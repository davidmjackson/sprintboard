# S6.4 — Complete sprint

## Story (backlog)

> ### S6.4 Complete sprint
> - **AC:** completing sets status complete.
> - **AC:** incomplete tickets return to the backlog (`sprint_id` null).
> - **AC:** completed tickets retain their Done status and sprint history.

The last story in E6's sprint lifecycle: future → (start, S6.3) → active → (complete, S6.4).

## What this is, and what it is not

A `future` sprint gets a **Start** button (S6.3). An `active` sprint gets a **Complete**
button (this story). Completing:

1. flips the sprint's `status` to `complete`;
2. returns its **incomplete** tickets (any `status != 'done'`) to the backlog by nulling their
   `sprint_id`;
3. leaves its **Done** tickets exactly where they are — `status` stays `done`, `sprint_id`
   still names the sprint. That retained `sprint_id` **is** the "sprint history" of AC3: a Done
   ticket in a completed sprint has a non-null `sprint_id`, which is precisely why S5.1's
   backlog rule (`sprint_id is null`) keeps it *out* of the backlog. AC3 and S5.1 are the same
   invariant read from two sides.

No new schema. No migration. The partial unique index (`sprints_one_active_per_project`), the
`sprints_id_project_unique` composite key, and the `tickets_sprint_fk … on delete set null
(sprint_id)` all pre-exist and are untouched. Completing does not free up the active slot for a
*new* start in the same click — that is the user's next action, and S6.3 already governs it.

## The central design decision: a two-table write with no transaction

Unlike S6.3 (one row, one table), completing touches **two tables** — `sprints.status` and
`tickets.sprint_id` — and Supabase's PostgREST gives the browser no client-side transaction
across two statements. Three ways to resolve this were considered.

**A — Client-sequenced, move-first (CHOSEN).** Two statements, in this order:

1. `update tickets set sprint_id = null where sprint_id = :id and status <> 'done'`
   (a single bulk UPDATE — atomic across all matching rows), returning the moved rows.
2. `update sprints set status = 'complete' where id = :id`, returning the row.

The **status flip runs last and is the commit marker**: because step 2 only runs if step 1
succeeded, a sprint that reads `complete` is *never* one whose incomplete tickets are still
attached. If step 1 succeeds and step 2 fails, the sprint stays `active` with its incomplete
tickets already in the backlog — an odd but **visible and self-correcting** state: the sprint
is still active, the user sees the error, and a retry re-runs step 1 (idempotent — the rows
are already null, Done tickets never matched) then step 2. Nothing is silently lost.

**B — Postgres RPC (`complete_sprint(id)`), truly atomic.** Rejected on two independent
grounds. First, it is a **schema change** (a new database object → a migration), which under
autopilot is a stop-and-ask. Second, it walks through a door CLAUDE.md holds shut at Rung 1:
"Direct React to Supabase … No backend API yet." Server-side transactional business logic is
exactly the Rung-2 FastAPI surface that is parked. The atomicity B buys is not worth
introducing the first piece of backend logic into a project whose whole thesis is a clean
front-end slice.

**C — Client-sequenced, status-first.** Rejected. Flipping status first fails *unsafe*: a
mid-operation failure leaves a `complete` sprint with incomplete tickets still attached —
AC2 silently violated, and those tickets are neither in the backlog (`sprint_id` not null) nor
in a running sprint. C's intermediate state is invisible and wrong; A's is visible and
recoverable.

**Why A is acceptable despite not being atomic.** There is no production data, a single owner
per project, and no concurrent-editor story at Rung 1. The realistic failure is a network blip
mid-operation, and A degrades to a visible, retryable, non-lossy state under exactly that. The
cost of true atomicity (B) is a parked architectural layer. This is the project's standing
trade — "premature generalisation is the real risk" — applied to one write.

> **Judgement call flagged for veto:** choosing client-sequencing (A) over an atomic RPC (B).
> A touches no schema and no security boundary, so it is not a stop-and-ask under autopilot; B
> would be. Recorded here so it can be overridden.

## Data layer — `completeSprint(id)` in `src/lib/sprints.ts`

```
export type CompleteSprintResult =
  | { ok: true; sprint: Sprint; returnedTickets: Ticket[] }
  | { ok: false; error: 'unknown' }
```

- **Single error tag `'unknown'`.** Unlike `startSprint`'s `already_active` (`23505` from the
  partial index), completing hits **no** user-correctable constraint: there is no unique index
  on `status = 'complete'`, so a second/third complete is legal, and setting an already-`done`
  ticket's `sprint_id` is a plain update. The only failures are non-correctable (network, or an
  RLS zero-row match on a cross-tenant/missing id). So one `'unknown'`, matching `createSprint`.
- **Step 1 uses `{ sprint_id: null } satisfies TicketUpdate`** (`sprint_id` is not among
  `TicketUpdate`'s omitted columns — the same field S6.2 wrote) and **`.select()`** to return
  the moved rows for the UI's local patch. No `.single()` — it is a set, possibly empty (a
  sprint of all-Done or zero tickets moves nothing; that is success, not failure).
- **Step 2 uses `.eq('id', id).select().single()`.** This is also the **RLS gate**: for a
  cross-tenant caller, step 1's bulk update filters to zero rows and returns *no error* (an
  UPDATE matching nothing is not an error), so it cannot be the gate. Step 2's `.single()` on a
  zero-row match **errors**, and we map that to `'unknown'` — never leaking existence, and never
  mutating A's sprint. Same RLS reasoning as `startSprint`, via `sprints_owner` / `tickets_owner`.
- **`returnedTickets` is authoritative, not a guess:** they are the database's own post-update
  rows (`sprint_id` now null), so patching them into the UI locally is as correct as a refetch
  for those rows, without touching the rest.

## UI — `CompleteSprintButton` + shell wiring

**`src/routes/CompleteSprintButton.tsx`** — mirrors `StartSprintButton`: self-contained
`pending`/`error`, owns the async, a pure view otherwise. Rendered by `SprintsTab` **only for
`active` sprints** (`sprint.status === 'active'`), exactly as Start renders only for `future`.
A single generic error message (`Something went wrong. Please try again.`) — there is no
user-correctable case to distinguish, so no second message. On success it calls **one**
callback carrying both results.

**Shell (`ProjectShell`)** gains `onSprintCompleted(sprint, returnedTickets)`, because
completing changes state in **two** of the shell's lists at once and the two must move together:

- replace the sprint in `sprintsLoaded` by id (identical logic to `onSprintUpdated`, guarded on
  `phase === 'loaded'` — the load-bearing guard from S4.6);
- in `loaded`, replace every ticket whose id is in `returnedTickets` with its returned row
  (`sprint_id` now null), same `phase === 'loaded'` guard.

This is a **local mutation, not a refetch** — the codebase's stated convention (three "local
mutation, not a refetch" comments in `ProjectShell`) and the reason `onRetry` is not reused
here: `onRetry` flips both tabs to `loading` (a flash on a deliberate action) and re-reads
sprints we just mutated, whereas the returned rows already carry the DB truth. A separate
`onSprintUpdated` + a ticket-batch callback would work but splits one atomic-to-the-user event
across two calls; a single `onSprintCompleted` keeps the sprint-row swap and the ticket moves
in one update so the count badge and the sprint status never render out of step.

The Sprints tab's per-sprint ticket **count badge** (S6.2) updates for free: it reads
`selectSprintTickets(tickets, sprint.id)`, and the moved tickets no longer name the sprint, so
the completed sprint's badge drops to its retained-Done count. The Backlog view (S5.1,
`sprint_id is null`) gains the returned tickets for free for the same reason. No change to
either; both are consequences of the shell's `tickets` list being patched.

Not in scope: the ticket-detail sprint picker (S6.2) will now list `complete` sprints as
options. Whether a completed sprint should be selectable there is a separate concern; this
story does not change picker behaviour, and completing does not crash it.

that shape; a step-1 error short-circuits to `'unknown'` **without** issuing step 2 (prove the
