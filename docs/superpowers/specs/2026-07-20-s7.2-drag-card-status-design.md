# S7.2 — Drag a card to change status

**Story:** E7 Board · S7.2. Delivered under `/autopilot` (self-approved design; none of the
four stop-and-ask exceptions — schema, security boundary, locked scope, unimplementable AC —
apply).

## Acceptance criteria
1. Dragging a card to a new column updates its status in the database.
2. The change survives a refresh.
3. An optimistic UI update reverts if the write fails.

## Context checked against the code before designing
- `updateTicket(id, patch)` already exists and already accepts `status` (`status` is in
  `TicketUpdate`, which omits only the trigger/immutable columns). **No data-layer change.**
- `status` is a `text` + check column (`todo|in_progress|in_review|done`), never an enum
  (CLAUDE.md). No schema change, no migration, no new RLS. The write reuses the existing
  owner-scoped `tickets_owner` policy, already proven live (S4.2 owner-update round-trip and
  cross-tenant rejection are in `tickets.integration.test.ts`). So this is **not** a
  security-boundary diff — an ordinary story, single strong adversarial reviewer, no ultracode.
- `ProjectShell.onTicketUpdated(ticket)` is the shared local-mutation reducer: it replaces one
  ticket by id in the shell's `loaded` list (guarded on `phase === 'loaded'`). It is the
  optimistic-apply / reconcile / revert hook the board will drive — the same hook the detail
  dialog uses.
- `TicketDetailDialog.commit()` is the precedent for an optimistic, field-scoped, roll-back-on-
  failure write. The board move follows it.
- The board (`BoardTab`) renders the **active sprint's** tickets (S7.1) into the four fixed
  columns from `TICKET_STATUSES`, each card a `<button>` (`TicketCard`) that opens the detail
  modal on click.

## Design

### Drag mechanism — native HTML5 drag-and-drop, no new dependency
`TicketCard` becomes `draggable` and gains `onDragStart` / `onDragEnd`. Each column `<section>`
gains `onDragOver` (which calls `preventDefault` so the browser will fire `drop`) and `onDrop`.

The dragged ticket's id travels through **React state** (`draggingId` in `BoardTab`), set on
`onDragStart` and cleared on `onDragEnd`/`onDrop` — **not** through `dataTransfer`. Reasons:
`dataTransfer` is not reliably constructed in jsdom (so a state payload is what makes the drag
testable without a live browser), and a state payload is robust in real browsers too.
`onDragStart` still calls `e.dataTransfer?.setData('text/plain', ticket.id)` best-effort, because
Firefox refuses to start a drag unless `setData` is called — but nothing reads it back; the
optional chaining keeps it a no-op under jsdom.

Rejected alternatives:
- **`@dnd-kit/core`** — a real dependency and a pointer-sensor/collision API for what is one
  card-to-column drag. YAGNI against the minimalist stack. Native DnD is enough.
- **`dataTransfer` as the payload** — finicky/unsupported in jsdom, forcing every test to
  hand-build a `dataTransfer` mock for no gain over a state payload.

The card is BOTH draggable and a click-to-open `<button>`. A drag and a click are distinct DOM
gestures, so both coexist: a click still opens the modal, a drag moves the card.

### Status write — optimistic, field-scoped, reverts on failure (mirrors `commit()`)
`BoardTab` owns a `moveTicket(ticketId, toStatus)` handler:

1. Find the ticket in the latest list. If `toStatus === ticket.status`, **no-op** — no write, no
   optimistic update (dropping a card back on its own column must not churn the DB).
2. Optimistic apply: `onTicketUpdated({ ...ticket, status: toStatus })`.
3. Persist: `const result = await updateTicket(ticketId, { status: toStatus })`.
4. On failure: revert **only the `status` field**, merged onto the **latest** ticket (read from a
   `ticketsRef`, see below): `onTicketUpdated({ ...latest, status: ticket.status })`. Show the
   error (below).
5. On success: reconcile `status` + the DB-refreshed `updated_at` onto the latest ticket:
   `onTicketUpdated({ ...latest, status: result.ticket.status, updated_at: result.ticket.updated_at })`.

**Latest-merge via `ticketsRef`.** `BoardTab` keeps a `ticketsRef` synced to the context `tickets`
in an effect (writing a ref during render is forbidden by the project's react-hooks/refs lint
rule — the detail dialog syncs `ticketRef` the same way). The revert/reconcile in steps 4–5 read
`ticketsRef.current` so they merge the `status` change onto whatever is freshest **now**, not onto
the ticket as it was when the drag started. This is deliberate: it means a concurrent edit to a
*different* field of the same ticket (e.g. a summary edit in the detail dialog while a drag is in
flight) is preserved rather than clobbered — the exact Ultracode-E "out-of-order optimistic saves
clobber" debt that memory flagged S7.2 as the place it would next bite. Scoped to the one field
each write owns, it does not resurrect that debt; it pays it down for `status`.

Deliberately deferred (same limitation the detail dialog documents): two drags of the *same* card
resolving out of order still last-resolved-wins. Two drags of *different* cards are independent
(different ids) and safe.

### Failure surface
A `moveError` state (`string | null`) in `BoardTab`, rendered as a `role="alert"` line above the
column grid: e.g. *"Could not move MP-1 to In Review. Please try again."* It is set on a failed
write and cleared at the start of the next drag and on a successful move. AC3 only requires the
revert; the message is UX parity with the detail dialog's inline error and tells the user why the
card jumped back.

### Data flow
`onDragStart(card)` → `draggingId = ticket.id` → user drags over a column (`onDragOver`
preventDefault) → `onDrop(column.status)` → `moveTicket(draggingId, column.status)` → optimistic
`onTicketUpdated` → board re-renders the card in its new column → `updateTicket` settles →
reconcile or revert. The board reads `tickets` from context, so the moved card appears in the new
column purely from the optimistic state update; nothing else re-fetches.

### What is NOT built (kept out on purpose)
- **No keyboard drag / no status dropdown.** Native DnD is mouse/touch only, and there is no
  keyboard status editor. The AC is "drag a card"; a keyboard-accessible status control is a
  later a11y story. Noted as a follow-up, not silently dropped.
- **No cross-sprint / backlog drag, no reordering within a column.** Only status changes. Column
  membership is still `status`, order is still `number` (S7.1).
- **No drop-target highlight animation** beyond what the AC needs; a hover/affordance polish story
  can add it.

## Testing
- **jsdom `BoardTab` tests** (drive drag with `fireEvent.dragStart(card)` then
  `fireEvent.drop(column)`, since `userEvent` has no drag API):
  - a drop on a different column optimistically calls `onTicketUpdated` with the new status AND
    calls `updateTicket` with `{ status }`;
  - on a mocked `updateTicket` success, the reconcile keeps the new status;
  - on a mocked `updateTicket` failure, `onTicketUpdated` is called a second time reverting to the
    original status, and the `role="alert"` error shows;
  - dropping on the SAME column is a no-op: `updateTicket` is not called;
  - a click still opens the detail modal (drag did not break click-to-open).
  - Prove the failure test bites by watching it fail with the revert removed.
- **Live integration test** (`tickets.integration.test.ts`, owner path): create a project +
  ticket, `updateTicket(id, { status: 'in_review' })` through the **app function**, re-read the
  row, assert `status === 'in_review'`. This proves AC1 (DB update) and AC2 (survives a re-read /
  refresh) against the real database and the real check constraint — not jsdom-only.
- **Whole-branch review** (most capable model, adversarial brief) + **security review** before PR.
- `npm run verify` run in full by the orchestrator; test-count tripwire checked (0 skipped).
