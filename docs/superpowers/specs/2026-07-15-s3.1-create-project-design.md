# S3.1 — Create project with a derived key (design)

**Story (E3 Projects):** create a project; the key is auto-derived from the name
(uppercase, 2–4 chars), editable before save; the key is validated against the format
rule and must be unique per owner; an invalid or duplicate key is rejected before save.

First feature that writes to the database through the browser client under
owner-scoped RLS.

## Architecture (three layers, each independently tested)

| Layer | File | Tested by |
|---|---|---|
| Key logic (pure) | `src/lib/project-key.ts` — `deriveProjectKey(name)`, `isValidProjectKey`, `PROJECT_KEY_PATTERN` (the exact mirror of the DB `projects_key_format` check). | `project-key.test.ts` (unit) |
| Data layer | `src/lib/projects.ts` — `createProject({ownerId,name,key})` → discriminated `CreateProjectResult`; maps Postgres `23505` to `duplicate_key`. | `projects.test.ts` (unit, mocked) + `projects.integration.test.ts` (**live**) |
| UI | `src/routes/CreateProjectDialog.tsx` — shadcn Dialog + Form; key auto-suggests until the user edits it; zod `CreateProjectSchema`; duplicate surfaces as a key field error. | `CreateProjectDialog.test.tsx` (component) |

Wired via a **New project** button in the authed header (`App.tsx`); it moves into the
left-nav in S3.2.

## Decisions

- **`owner_id` is the caller's `auth.uid()`** — the RLS insert policy is the boundary,
  not app code. The live test proves the insert succeeds and the constraints bite.
- **Duplicate key is a result, not a throw.** It is an expected, correctable outcome;
  `createProject` returns `{ok:false, error:'duplicate_key'}` and the dialog shows a
  field-level message rather than crashing.
- **Validate at both edges** (CLAUDE.md): zod (`CreateProjectSchema`) on the client,
  the `projects_key_format` + `projects_owner_key_unique` constraints in the DB. The
  client pattern is *identical* to the DB check, and a test pins them together.
- **Key derivation is a suggestion, never authoritative.** Multi-word → initials,
  single word → first chars; the user can always override, so it only needs to be a
  good guess. `Sprintboard` → `SPR` (the backlog's "SPB" is illustrative of the format).
- **The live test drives a signed-in RLS client, not the app singleton**, because the
  app's client is unauthenticated in the test process — same split the signup suite uses
  (live proves the DB contract; unit proves `createProject`'s error mapping).

## Not in this story

On success the dialog just closes — there is **no project list to refresh yet** (S3.2)
and **no project view to navigate to** (S3.3). An `onCreated` callback is exposed for
S3.2 to consume.
