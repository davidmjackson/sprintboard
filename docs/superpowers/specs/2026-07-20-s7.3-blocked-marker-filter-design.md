# S7.3 ŌĆö Blocked marker and blocked-only filter (design)

Closes **E7 Board**. Story labels: `board`.

## Acceptance criteria (from backlog)

- **AC1:** blocked cards show a clear marker and their reason on hover or open.
- **AC2:** a filter toggles the board to show blocked cards only.

## What already exists (checked against the code, not assumed)

- **The marker exists.** `TicketCard` renders `BlockedBadge` whenever `ticket.is_blocked`
  (S4.4). The badge shows a `Ban` icon + the visible word "Blocked".
- **The reason on _open_ exists.** `TicketDetailDialog` renders a `role="status"` banner with
  `blocked_reason` and `blocked_since` when the ticket is blocked (S4.4).
- **`is_blocked` / `blocked_reason` are already-read columns.** `listTickets` selects all
  columns; the board receives full ticket rows through `ProjectShellContext`. No new read.

So of AC1, only the **reason on _hover_** is missing, and AC2's filter is entirely new.

## Scope guard

Pure client-side view logic. **No** schema, migration, constraint, RLS, secret, data-layer,
or write change. "Blocked is a flag, never a board column" (CLAUDE.md) is preserved: the
filter narrows _which_ cards show; it never adds a column or moves a card. Nothing here touches
a security boundary or a parked feature.

## AC1 ŌĆö reason on hover

Add an optional `reason?: string | null` prop to `BlockedBadge`. When present, set the badge's
native `title` to `Blocked: <reason>`; otherwise `title="Blocked"`. `TicketCard` passes
`ticket.blocked_reason`. Hovering the marker surfaces the reason without opening the ticket.

**Why `title`, not a custom tooltip:** a native `title` is zero-dependency, keyboard/SR-adjacent,
and directly assertable in jsdom (`toHaveAttribute('title', ŌĆ”)`). A Radix/shadcn tooltip component
does not exist in the repo and would be new surface for a one-line affordance ŌĆö YAGNI. The badge
already carries the visible "Blocked" text for screen readers (the S4.4 decision), so `title` is
additive detail, not the sole affordance.

**Why the badge, not the whole card:** the reason _explains the marker_, so the tightest mapping
is markerŌåÆreason. The card is a button whose accessible name is the summary; overloading its
`title` with block state would muddy that.

## AC2 ŌĆö blocked-only filter

- **Control:** a native `<input type="checkbox">` with an associated `<label>` reading
  "Blocked only", in a small toolbar row above the column grid. Native, not a shadcn component
  (none exists; native form controls are the jsdom-friendly repo convention, cf. the sprint
  `<select>`). Assertable via `getByRole('checkbox', { name: /blocked only/i })`.
- **State:** local `useState(false)` in `BoardTab` (`blockedOnly`). Ephemeral view state ŌĆö a
  filter is not persisted in Phase 1 (YAGNI; no URL/storage sync).
- **Rendered only when there is an active sprint.** With no active sprint there is nothing to
  filter, and the no-active-sprint caption owns that state. Keeps the toolbar out of the empty
  state.
- **Predicate lives in `board.ts`**, as `selectBlockedTickets(tickets)` ŌåÆ `is_blocked` rows only.
  This mirrors the component's existing shape: `BoardTab` composes `selectActiveSprint` Ōłś
  `selectSprintTickets` and inlines no ticket-selection predicate. The one place "blocked is a
  flag" gets read for filtering is a named, unit-tested selector, consistent with its siblings.
  It is a one-liner, but so is `selectSprintTickets`'s membership rule; parity beats a lone inline
  `.filter`.
- **Application:** `const visible = blockedOnly ? selectBlockedTickets(boardTickets) : boardTickets`,
  then the existing per-status column split runs over `visible`. Drag (S7.2) is unaffected ŌĆö a
  blocked card can still be dragged while filtered.

### Empty state under the filter

When the filter is on and a column has no blocked cards it still shows the existing
"No tickets yet." The checked "Blocked only" box directly above the grid is the context that
explains empty columns, so no new all-empty caption is added (YAGNI; can be a later UX polish if
a reviewer wants it). This is a deliberate accept, recorded here.

## Testing

Client/unit only ŌĆö **no live integration test**. This story adds no DB contract: the reads it
uses (`is_blocked`, `blocked_reason`) are already covered live by S4.4. A live test here would
re-cover an existing read and spend the auth-flake budget for nothing. (Consequence: the
`full ŌłÆ test:unit` live-gap tripwire stays **60** ŌĆö every test added here raises both counts
equally.)

Coverage, each proven to bite by breaking the code:

1. `BlockedBadge`/`TicketCard`: a blocked card's marker has `title="Blocked: <reason>"`; with no
   reason, `title="Blocked"`. (Break: drop the prop ŌåÆ title reverts, test fails.)
2. `selectBlockedTickets`: returns only `is_blocked` rows; `[]` when none. (Break: `!t.is_blocked`.)
3. `BoardTab`: the "Blocked only" checkbox is present when a sprint is active and **absent** when
   not (positive + negative control); toggling it on hides an unblocked card and keeps a blocked
   one; toggling off restores. (Break: ignore `blockedOnly` ŌåÆ the hide assertion fails.)

## Decisions log (autopilot self-approved)

- Reason-on-hover via native `title` on the badge, not a tooltip component or the card. ŌĆö YAGNI + tight mapping.
- Filter predicate as `selectBlockedTickets` in `board.ts`, not inline. ŌĆö parity with sibling selectors.
- Native checkbox, local ephemeral state, active-sprint-only. ŌĆö jsdom convention + nothing to filter otherwise.
- No all-empty "no blocked tickets" caption. ŌĆö checkbox context suffices; YAGNI.
- No live integration test. ŌĆö no new DB contract; keeps live-gap tripwire at 60.
