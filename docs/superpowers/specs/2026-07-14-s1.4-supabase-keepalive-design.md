# S1.4 — Supabase keepalive heartbeat

**Date:** 2026-07-14
**Story:** S1.4, epic E1 (foundation)
**Status:** approved, ready for implementation

## Problem

The Supabase free tier pauses a project after roughly 7 days without activity. A
paused project used to be an inconvenience. It is now load-bearing: S1.3's live
RLS isolation suite runs against the real database, and S1.5 made `npm run verify`
a required status check with no bypass actors. So a paused database does not
merely break the app — **it blocks every merge, for everyone, including a PR whose
only purpose is to fix it.**

Nothing was keeping the database awake. The cron job that CLAUDE.md described as
live did not exist.

## What actually keeps Postgres awake

Measured against the live project (`xcnmyhozmcopcpxlagrk`) rather than assumed:

| Endpoint | Anon key result | Reaches Postgres? |
| --- | --- | --- |
| `GET /rest/v1/` | **401** — "Only the `service_role` API key can be used" | — |
| `GET /rest/v1/tickets?select=id&limit=1` | **200**, body `[]` | Yes |
| `GET /auth/v1/health` | 200 | No — GoTrue's own health check |

Two findings drive the whole design.

**The obvious endpoint is a trap.** `/rest/v1/` is what a person reaches for, and
it 401s for `anon`. Making it work would mean putting the service-role key into a
third-party cron service — a direct violation of the project's first security
rule. It must not be used, and the reason is recorded here so nobody rediscovers
it by trying.

**A 200 is not sufficient evidence of life.** A cached edge response would return
200 forever while the database slept. The `tickets` request is provably not cached:
the live test run returned `CF-Cache-Status: DYNAMIC` with
`x-envoy-upstream-service-time: 103` — 103ms of real upstream work. RLS is
evaluated *inside* Postgres, so filtering an anonymous caller down to zero rows is
itself proof the database woke up and ran the query. The empty array is the
success signal, not an error.

## Design

**The keepalive is cron-job.org. The repository's job is to assert the contract
the cron depends on.**

This split is the core of the design. Nothing in the repo needs to *be* a
keepalive — the cron pings daily whether or not anyone is working. What the repo
must prevent is the contract rotting underneath the cron, which is precisely what
already happened to `/rest/v1/`. **A cron pointed at a dead contract runs green
forever and keeps nothing alive.** That failure is silent, and it surfaces weeks
later as a paused database that blocks all merges.

### The cron (external, configured 2026-07-14)

| Field | Value |
| --- | --- |
| URL | `https://xcnmyhozmcopcpxlagrk.supabase.co/rest/v1/tickets?select=id&limit=1` |
| Method | `GET` |
| Header | `apikey: <VITE_SUPABASE_ANON_KEY>` |
| Schedule | Daily, 06:00 UTC |
| On failure | Email notification enabled |

Verified by cron-job.org's own test run: `200 OK`, `Content-Length: 2`,
`Content-Location: /tickets?limit=1&select=id`, `sb-project-ref:
xcnmyhozmcopcpxlagrk`. This satisfies AC-2 against the real configured job, not
against a hand-run curl — a curl proves the URL works, not that the job points at
it correctly.

**Daily, not the backlog's "every 2 to 3 days".** Supabase pauses at ~7 days. On a
2-day schedule, three consecutive silent misses reach 6 days — one more and every
merge is blocked. Daily requires six consecutive failures to reach the same cliff.
The cost is one trivial GET per day. The backlog wording is superseded; CLAUDE.md
documents the schedule that is actually configured.

**Redundancy is the failure email, deliberately.** A GitHub Actions scheduled
workflow was considered as a backup heartbeat and rejected: GitHub disables
scheduled workflows after 60 days of repository inactivity — exactly the
quiet-portfolio scenario the keepalive exists for. It would be a backup that dies
precisely when it is needed. cron-job.org's own failure notification is the right
mechanism: the monitor reports its own death.

### The contract test (in-repo)

`src/test/keepalive.integration.test.ts` — a real live GET against the same URL
with the same anon key, asserting:

1. status is `200`; **and**
2. the body is a **JSON array**.

The second assertion carries the weight. A 200 alone can come from a cache or a
gateway; an array proves PostgREST returned a result set from Postgres. It is the
in-code form of the `CF-Cache-Status: DYNAMIC` reasoning above.

It joins the existing live suite, so **CI runs it on every PR**. If Supabase ever
revokes anon access to this path the way it did to `/rest/v1/`, a PR goes red with
a clear message — instead of the database silently pausing weeks later.

`npm run keepalive` runs that single test file. **The manual trigger and the
automated check are the same act**: running it performs a real ping and asserts the
result. Two artefacts that could drift would be worse than one that cannot.

### Credentials

No new secrets. `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` are already
GitHub Actions variables — both are public by design and ship in the browser
bundle. The service-role key is not involved anywhere in this design.

The S1.3 credential guard is reused: a missing key **fails red in CI, never
skips**. A skipped heartbeat check reports a liveness it has not established —
the same principle that governs the RLS suite.

## Consequences

- **Test count moves 43 → 44.** The standing heuristic "43 = real, 30 = the RLS
  suite silently skipped" must be updated, or a correct run will be misread as a
  broken one.
- **CLAUDE.md's Infrastructure section** gets the real schedule and endpoint,
  replacing the aspirational description of a job that did not exist.
