# R2.2 — Grounded estimation assistant (SPRIN-41)

**Status:** approved (autopilot self-approval, 2026-07-21)
**Epic:** SPRIN-40 "Rung 2 — Grounded AI layer" · **Story:** SPRIN-41
**Predecessors:** R2.0 (#38, walking-skeleton decomposition), R2.1 (#39, traceable decomposition)

## 1. Why

The project charter is "a credible Jira-style board with a **grounded-AI layer** added at Rung 2."
R2.0 gave the AI epic→stories decomposition; R2.1 made it traceable (coverage-gap / scope-creep).
R2.2 is the last planned Rung 2 slice: **grounded estimation** — each proposed child work item
arrives with a defensible story-point estimate rather than an unsized ticket.

Both prior specs explicitly parked estimation here and teed up the cheap path:

> "R2.2 (grounded estimation) plugs into the same `llm.py` seam and can add estimate fields per
> proposal without reshaping this contract." — R2.1 spec §8

This slice walks through that door and no further.

## 2. Scope

**In:**
- The model estimates each proposed item on a **fixed Scrum scale** and gives a one-line size reason.
- A new pure module `api/app/estimation.py::analyze` **validates** each estimate onto the scale
  (snapping off-scale values deterministically) and **derives** an epic rollup total.
- The `/decompose` response carries the new fields, additively.
- The React decompose panel shows a `{n} pts` chip + reason per proposal and an "Estimated total"
  summary line.
- On accept, each created ticket's estimate is **persisted into the existing `story_points` column**
  via the existing `createTicket({ storyPoints })` path.

**Out (deferred, YAGNI for this slice):**
- Editing an estimate before accepting (whole-proposal editing was deferred at R2.0; unchanged here).
- A standalone "estimate this existing ticket" endpoint — estimation rides the decomposition flow only.
- Selection-reactive running total in the UI (the summary shows the model's full-set rollup; a
  live "selected total" is a later UX nicety, noted below).
- Historical-velocity grounding (would require the service to read the DB — ruled out by the
  stateless decision). Grounding here is scale + reason + deterministic rollup, not team history.
- Any schema/RLS/migration change. `story_points` is an existing first-class column.

## 3. Constraints honoured (from CLAUDE.md and memory)

- **Stateless service** — estimation adds no DB access and no secret; `/decompose` still only verifies
  the JWT (authN) and calls Claude. Not a security-boundary diff (auth/RLS/secret/CI-gate all untouched).
- **`llm.propose` is the single mock seam** — CI stays deterministic and needs no `ANTHROPIC_API_KEY`.
- **Model maps, Python derives** — the model proposes numbers; Python owns the scale validation and the
  rollup, so the derivation is deterministic and exactly testable and the model can't contradict itself
  (the R2.1 lesson: never let the model emit a total that disagrees with its parts).
- **`verify.yml` byte-unchanged**; `api.yml` stays a separate non-required check.
- **No Postgres ENUM, no inlined column names, core fields stay real columns** — untouched; we *use*
  the existing `story_points` column, we don't reshape anything.

## 4. Data flow (unchanged topology; additive payload)

```
Epic detail dialog                 POST /decompose  (JWT verify → Claude, structured out)
  "Decompose with AI"  ──────────▶   llm.propose(epic)  → proposals[] each with
                                       {title, description, type, rationale, covers,
                                        estimate, estimate_reason}
                                     traceability.analyze(deliverables, proposals)  → gaps/creep
                                     estimation.analyze(response)  → snap estimates, sum → estimate_total
  proposals panel      ◀───JSON────  { proposals[], coverage_gaps[], scope_creep[], estimate_total }
    per item: "{n} pts" chip + reason
    summary: "Estimated total: {N} pts"
  accept (all/subset)
      └─▶ createTicket({ …, storyPoints: p.estimate })  ──▶ Supabase (RLS, story_points)
```

## 5. The estimation module (`api/app/estimation.py`)

Pure, deterministic, and layered *after* `traceability.analyze` so each module owns one concern.

```python
ALLOWED_POINTS = (1, 2, 3, 5, 8, 13)   # modified Fibonacci; the only sizes the UI can show

def _snap(value: int) -> int:
    """Nearest allowed point by absolute distance; ties resolve to the smaller size.
    A model that emits 0, 7, or 100 is corrected here rather than surfacing an off-scale number."""

def analyze(response: DecomposeResponse) -> DecomposeResponse:
    """Snap every proposal's estimate onto ALLOWED_POINTS and derive estimate_total as the
    deterministic sum of the snapped estimates. Pure — the total is a function of the parts,
    never a second number from the model."""
```

- **Why snap, not reject:** rejecting would need an error path for a soft, advisory feature; snapping
  keeps the walking-skeleton honest and the UI always shows a valid Scrum size. The snap rule is
  "nearest allowed by absolute distance, ties resolve to the smaller size", documented and unit-tested:
  `0→1`, `4→3` (|4−3|=|4−5|=1, tie → smaller), `6→5`, `7→8` (|7−5|=2 vs |7−8|=1), `12→13`, `100→13`,
  negative → 1.
- **Why layer after traceability:** `traceability.analyze` already returns a `DecomposeResponse` with
  sanitised proposals; `estimation.analyze` takes that response and returns an augmented one. Composes
  as `estimation.analyze(traceability.analyze(deliverables, proposals))` in `decompose.py`. Each module
  stays independently testable.

## 6. Schema / contract changes (additive)

`api/app/schemas.py`:
- `Proposal` gains `estimate: int = 0` and `estimate_reason: str = ""`.
- `DecomposeResponse` gains `estimate_total: int = 0`.

`api/app/llm.py`:
- `PROPOSALS_SCHEMA` items gain `estimate` (integer) and `estimate_reason` (string), both `required`
  (structured-output rule: every property required, `additionalProperties:false`, no numeric bounds —
  so the scale is enforced in Python's `_snap`, **not** in the JSON schema).
- System prompt gains: size each item on the Scrum scale 1/2/3/5/8/13; give a one-line reason naming
  the main size driver (uncertainty, scope, or dependencies).

`src/lib/ai.ts`:
- `DecomposeProposal` gains `estimate: number` and `estimate_reason: string`.
- `DecomposeResult` (ok) gains `estimate_total: number`.
- Defensive defaults for forward/back-compat, exactly like `covers`: a response missing `estimate`
  → treat as absent (chip hidden), missing `estimate_total` → `0`. Keep snake_case wire names.

## 7. UI (`src/routes/TicketDetailDialog.tsx`)

- **Per proposal:** a `{estimate} pts` chip in the existing chip row (beside the coverage / creep
  chips), and `estimate_reason` as a muted `text-xs` line under the rationale. Hidden when the estimate
  is absent (forward-compat).
- **Panel summary:** add "Estimated total: {estimate_total} pts" next to the existing
  "Covers X of Y deliverables" line.
- **Accept:** the existing `acceptSelected` loop passes `storyPoints: p.estimate` into `createTicket`.
  Everything else about accept (clear-panel-after-attempt, partial-failure message) is unchanged.

The panel already re-derives on a deliverables edit (R2.1's `writeDeliverables` invalidation drops the
whole decomposition); estimates ride along in that same `proposals` state, so no new staleness seam is
introduced — the estimate is part of the same object that already gets nulled.

## 8. Testing

- **`api/tests/test_estimation.py` (new):** `_snap` table (0→1, 4→3, 6→5, 7→8, 12→13, 100→13, negative→1);
  `analyze` sums snapped estimates; empty proposals → total 0; a single off-scale estimate is snapped in
  the returned proposal *and* reflected in the total (proves snap happens before sum).
- **`api/tests/test_decompose.py` (extend):** the mocked `llm.propose` now returns estimates; assert the
  endpoint response carries per-proposal estimates and the derived `estimate_total`; an off-scale mock
  estimate is snapped in the response.
- **`api/tests/test_llm.py` (extend):** `PROPOSALS_SCHEMA` includes the two new required properties.
- **`src/lib/ai.test.ts` (extend):** parses estimates + total; forward-compat (missing fields →
  estimate absent, total 0); malformed guard still degrades to `request_failed`.
- **`src/routes/TicketDetailDialog.test.tsx` (extend):** panel renders the pts chip, the reason, and the
  total; **accept persists the estimate** — assert `createTicket` is called with `storyPoints` equal to
  the proposal's estimate (break-the-code check: drop the prop and watch it fail).

## 9. Review

Ordinary story (extends an existing endpoint's response + an existing write path with one field). **One
strong adversarial whole-branch reviewer + the standard security reviewer**, not the multi-agent deep
pass — it touches no security boundary (JWT verify, RLS, secrets, CI gate all unchanged). The security
reviewer's job is to confirm that non-boundary claim holds: that persisting `story_points` rides the
existing owner-scoped `createTicket` path and opens no new write surface.

## 10. The one judgement call (flag for veto)

**Persist the estimate into `story_points` on accept**, vs R2.1's display-only stance. Chosen because
`story_points` is a real first-class column, `createTicket` already accepts it, and an estimation
assistant whose numbers evaporate on accept is theatre. It is additive, uses the existing owner-scoped
write path, and changes no schema. If vetoed, the fallback is display-only (drop the `storyPoints` arg
and its accept test) — a one-line change.

## 11. Forward compatibility

- Contract stays additive; a later slice can persist the *whole* estimate trace (reason per ticket) or
  add selection-reactive totals without reshaping this response.
- `estimation.py` being a pure post-processor means richer grounding later (historical velocity when/if
  the service ever reads the DB at Rung 3, or a confidence band) is an isolated change.
- The Scrum scale lives in one constant (`ALLOWED_POINTS`) — a Rung-3 configurable estimation scale is a
  one-place change, not a sweep.
