# S3.3 — Project shell with Board and Backlog tabs

**Epic:** E3 Projects · **Jira:** SPRIN-21 · **Branch:** `s3.3-project-shell`

## Story

Inside an open project, show a **Board** tab and a **Backlog** tab. This is the
project shell; it replaces the `ProjectView` placeholder shipped in S3.2.

## Acceptance criteria

- An open project shows a Board tab and a Backlog tab.
- The shell renders even when the project has no tickets yet.

## Scope

UI and routing only. No ticket data exists until E4, so both tabs are structural
scaffolding. This is **not** a security-boundary change — no auth, RLS, secret, or
CI-gate surface is touched — so it takes the standard peer + security review, not the
deep multi-agent pass.

**Parked (E4+), explicitly not built here:** ticket loading, create-ticket,
drag-and-drop, sprint context. If a step seems to need one of these, stop.

## Design

### Routing (nested)

The tab selection lives in the URL and survives a refresh, matching the S3.2
convention that navigation state is URL-addressable. `ProjectView.tsx` is renamed to
**`ProjectShell.tsx`** and becomes the shell:

```
/projects/:projectId          -> ProjectShell; index route redirects to "board"
/projects/:projectId/board    -> <BoardTab/>
/projects/:projectId/backlog  -> <BacklogTab/>
```

`ProjectShell` keeps the existing guard unchanged:

- while the layout's project list is loading → a loading state;
- a `:projectId` not in the RLS-scoped list (another owner's, or deleted) → `Navigate`
  home. Nothing is revealed.

It renders the header (key + name), the tab bar, and an `<Outlet>`. It hands the
resolved project to its tabs via a nested outlet context
(`ProjectShellContext = { project: Project }`). The tabs do not need the project yet;
E4 will, and providing it now keeps that door open cheaply.

The tab bar is two **styled `NavLink` pills** (`to="board"` / `to="backlog"`,
relative paths) — not the shadcn Tabs primitive, because the selection *is* a route.
No new dependency; `NavLink` gives active styling and `aria-current` for free.

### Components

- **`BoardTab.tsx`** — renders the four fixed columns from `TICKET_STATUSES`, each
  with its display label and an empty state. A credible, empty board.
- **`BacklogTab.tsx`** — an empty-state message ("No tickets yet.").

### `domain.ts`

Add `TICKET_STATUS_LABELS: Record<TicketStatus, string>`:

| status | label |
|---|---|
| `todo` | To Do |
| `in_progress` | In Progress |
| `in_review` | In Review |
| `done` | Done |

CLAUDE.md forbids inlining the four column names in a component, a filter, or a
badge-colour map — this map is their single, correct home. Typing it as
`Record<TicketStatus, string>` makes it exhaustive by construction; a `domain.test.ts`
assertion also confirms every runtime status has a label.

## Tests (written first, from the ACs)

New **`ProjectShell.test.tsx`** (React Testing Library + `MemoryRouter`, mirroring
`AppLayout.test.tsx`):

- Opening a project shows both a Board and a Backlog tab link.
- The index route defaults to Board; the four column labels render.
- Clicking Backlog renders the backlog content.
- A `/projects/:id/backlog` deep link is active on load (survives a refresh).
- The shell renders with zero tickets without crashing (covers both ACs).
- A `:projectId` not in the list sends the user home.

Update **`AppLayout.test.tsx`**: point `renderApp` at the nested route shape and
replace the two assertions that reference the removed `"Board and Backlog arrive"`
placeholder.

Add to **`domain.test.ts`**: `TICKET_STATUS_LABELS` has an entry for every value in
`TICKET_STATUSES`.
