# S2.2 — Login, logout, session persistence (design)

**Story (E2 Authentication):** a user logs in with valid credentials, sees an error on
invalid ones; the session survives a page refresh; logout clears the session and
returns to the login screen.

Most of the machinery already exists from the S2.1 foundation. This story hardens the
`LoginPage` stub, adds logout, and proves persistence.

## What already exists (S2.1)

- `LoginPage` calls `supabase.auth.signInWithPassword` and shows a **generic** "Invalid
  email or password." on any failure, navigating to `/` on success.
- `AuthProvider` restores the session on mount via `getSession()` and tracks
  `onAuthStateChange`; `supabase.ts` sets `persistSession: true`. **Session persistence
  is therefore already implemented** — refresh = remount = `getSession()` reads the
  persisted session.
- `RequireAuth` redirects to `/login` whenever the resolved session is null.

## Decisions

- **Keep the generic login-error message.** Not distinguishing "wrong password" from
  "no such account" is deliberate — it avoids account enumeration. (Same posture as
  the signup duplicate discussion.)
- **Logout is the only new code.** A `signOut()` trigger, following the established
  pattern: the component calls `supabase.auth.signOut()` directly, `onAuthStateChange`
  nulls the session in context, and `RequireAuth` redirects to `/login`. No explicit
  navigate — the guard does it. `AuthProvider` stays state-only.
- **Logout affordance** lands on the authed placeholder screen (the only authed
  surface today): a minimal header showing the signed-in email + a "Log out" button.
  It moves into the real app shell at S3.3.

## Testing notes

- The logout-flow test captures the `onAuthStateChange` callback in the mock and has
  `signOut` invoke it with `(’SIGNED_OUT’, null)`, so the test exercises the real
  redirect path rather than asserting the call in isolation.
- Persistence is proven at the unit level (mount restores a persisted session). True
  browser-refresh survival is an E2E concern owned by E8 (S8.1).
- The live login test reuses the two confirmed RLS users — no new users, no dependence
  on the email-confirmation setting.
