# R2.1 — Traceable epic decomposition (coverage-gap & scope-creep detection)

**Date:** 2026-07-21
**Rung:** 2 (un-parked 2026-07-20). Second Rung 2 slice, building directly on R2.0 (#38).
**Status:** design approved (display-only + model-maps/Python-derives forks approved by David;
UI and testing decisions made under autopilot autonomy and recorded here).

---

## 1. Problem & goal

R2.0 shipped a walking-skeleton AI epic decomposition: the local `api/` service turns an epic's
`summary`/`context`/`deliverables` into child-work proposals, and each proposal carries a free-text
`rationale` that is **display-only** — a deliberate "forward-nod to R2.1."

R2.0 is a generic LLM wrapper. R2.1 makes it **defensible**: the AI shows *which* deliverable each
proposed story serves, and the tool then computes — deterministically, not as a second model opinion
— what the proposal set **misses** (coverage gaps) and what it **adds beyond the epic** (scope
creep). This is the portfolio differentiator named in CLAUDE.md's Rung 2 scope: *"AI epic
decomposition (traceable, with coverage-gap and scope-creep detection)."*

**Key insight — one mechanism, three views.** If each proposal declares which deliverables it serves,
then traceability *is* that mapping, coverage-gap = deliverables no proposal maps to, and scope-creep
= proposals mapping to nothing. R2.1 is therefore a single coherent slice, not three features.

## 2. Scope

**In scope**
- The `/decompose` response gains a per-proposal `covers: number[]` (indices into the epic's
  `deliverables`) plus two derived response fields: `coverage_gaps` and `scope_creep`.
- A pure Python derivation module computes gaps/creep from the mapping and sanitises `covers`.
- The React decompose panel surfaces the trace: a coverage summary, an uncovered-deliverables
  callout, per-proposal "delivers" chips, and a soft "not tied to a deliverable" flag.

**Out of scope (recorded, deferred)**
- **Any persistence of the trace.** Display-only — David's call. No schema change, no migration, no
  RLS change. Accept still persists only `title/description/type + parent_epic_id` via `createTicket`,
  exactly as R2.0. (This keeps R2.1 clear of every autopilot stop-and-ask boundary.)
- **Grounding against the epic's prose `context`.** Deliverables are the checkable unit; a proposal
  serving only the context surfaces as a *soft* creep flag, not a hard "out of scope" (see §4).
- Editing proposals before accept (R2.0-deferred); grounded estimation (R2.2); cloud deploy.

## 3. Data contract (`/decompose`)

All changes are **additive** — no existing field changes shape or meaning.

```
Proposal:            { title, description, type, rationale, covers: number[] }   // covers is NEW
CoverageGap:         { index: number, deliverable: string }
ScopeCreep:          { proposal_index: number, title: string }
DecomposeResponse:   {
  proposals:     Proposal[],       // covers on each is the SANITISED list (see §4)
  coverage_gaps: CoverageGap[],    // derived
  scope_creep:   ScopeCreep[],     // derived
}
```

- The model produces **only** the mapping (`covers`), because matching a story to a deliverable is
  genuine judgement. `coverage_gaps` and `scope_creep` are **never** asked of the model — they are
  derived in Python so the analysis cannot contradict its own mapping and is exactly testable.
- `covers` is added to `PROPOSALS_SCHEMA` in `llm.py` (`{"type":"array","items":{"type":"integer"}}`),
  with `additionalProperties:false` + `required` maintained; no numeric/length constraints (the
  structured-output limitation from R2.0 still holds). The system prompt is extended to instruct the
  model to cite deliverable indices (0-based) each item serves, and to leave `covers` empty for an
  item not tied to any listed deliverable.

## 4. Derivation (pure, deterministic) — `api/app/traceability.py`

A new FastAPI-free module, mirroring how `llm.py` stays framework-free — so it is unit-testable in
isolation and `decompose.py` stays thin.

```
analyze(deliverables: list[str], proposals: list[Proposal]) -> AnalysisResult
    # AnalysisResult carries: proposals (with sanitised covers), coverage_gaps, scope_creep
```

**Rules (decided; the edge cases are the whole point of deriving in Python):**
1. **Sanitise `covers` first.** For each proposal keep only indices `0 <= i < len(deliverables)`,
   de-duplicated and sorted ascending. This is what goes into the response, so client and server
   agree on the trace. Out-of-range indices (a hallucinated index) are silently dropped — if that
   empties a proposal's `covers`, it correctly becomes creep. (Sanitising is intentionally *not*
   surfaced as its own signal; a dropped index simply doesn't count as coverage.)
2. **`coverage_gaps`** = every deliverable index referenced by *no* sanitised `covers`, in order,
   each with its deliverable text.
3. **`scope_creep`** = every proposal whose sanitised `covers` is empty — i.e. not tied to any
   *listed deliverable*. Surfaced as a **soft "review — not tied to a deliverable" flag, not
   "out of scope"**: a story can legitimately serve the epic's prose `context`; the user judges,
   with the model's existing `rationale` as the explanation.
4. **Zero-deliverable epic** (`deliverables == []`): analysis is vacuous. Return **empty**
   `coverage_gaps` **and empty** `scope_creep` — with no deliverables there is nothing to trace to,
   and flagging every proposal as creep would be pure noise. The UI shows a "no deliverables to trace
   against" note instead.

`decompose.py` calls `traceability.analyze(...)` after `llm.propose(...)` and assembles the response.
The existing `LLMError -> 502` mapping is unchanged; `analyze` is pure and cannot raise on valid
`Proposal` objects.

## 5. UI (`src/lib/ai.ts` + `TicketDetailDialog.tsx`) — decided under autopilot

**`ai.ts` (`decomposeEpic`).** `DecomposeProposal` gains `covers: number[]`. `DecomposeResult.ok`
gains `coverage_gaps: {index, deliverable}[]` and `scope_creep: {proposal_index, title}[]`. The
existing response guard is extended defensively: if `coverage_gaps`/`scope_creep`/`covers` are absent
or not arrays, default each to `[]` rather than failing the whole call — a forward-compatible service
that omitted them should still decompose (the panel just shows no trace). `Array.isArray(proposals)`
stays the load-bearing guard that gates `ok: true`.

**Decompose panel (the epic-only block, dialog lines ~757–832).** Display-only; selection/accept
logic untouched. Additions:
- **Coverage summary** above the proposal list: `Covers N of M deliverables` (M = total, N = covered).
  Omitted when the epic has no deliverables; a "No deliverables to trace against" line shows instead.
- **Uncovered-deliverables callout** when `coverage_gaps` is non-empty: a small muted list, each item
  the deliverable text, headed e.g. "Not covered by any proposal". Advisory only — it does **not**
  block or alter accept.
- **Per-proposal "delivers" chips:** under each proposal's rationale, one small chip per covered
  deliverable showing the deliverable text (truncated). A proposal in `scope_creep` shows a single
  soft-amber "Not tied to a deliverable" chip instead. Purely informational; the checkbox/accept
  behaviour is unchanged, and creep proposals remain selectable (the user may knowingly accept them).
- Gaps/creep are **advisory**: a user may accept a subset that leaves a gap, or accept a creep item;
  R2.1 never gates the write on the analysis (matches display-only).

Rationale for these choices: they reuse the existing panel layout and the `TICKET_TYPE_LABELS` /
muted-text idiom already in the dialog; no new component files, no new dependency, no new state beyond
reading the two new result fields. Chips over a table keep it inside the narrow main column.

## 6. Testing

**Python (`api/tests/`), model stubbed via the `llm.propose` seam — no key, no network:**
- **`test_traceability.py` (new), the core of R2.1** — pure-function coverage of `analyze`:
  full coverage → no gaps/creep; an uncovered deliverable → one gap; a proposal with empty `covers`
  → one creep; an out-of-range index → dropped, doesn't count as coverage, empties→creep; duplicate
  indices → de-duped; zero-deliverable epic → empty gaps AND empty creep; sanitised `covers` is
  sorted/unique on the returned proposals.
- **`test_decompose.py`** — endpoint returns `coverage_gaps`/`scope_creep` matching the derivation
  for a stubbed proposal set; `covers` present on each proposal; `LLMError -> 502` still holds.
- **`test_llm.py`** — `PROPOSALS_SCHEMA` includes `covers`; a stubbed model response with `covers`
  parses into `Proposal`s.

**React (Vitest), `fetch` mocked:**
- **`ai.test.ts`** — parses `covers`/`coverage_gaps`/`scope_creep`; missing new fields default to
  `[]` without failing the call; a non-array `proposals` still degrades to `request_failed`.
- **`TicketDetailDialog.test.tsx`** — renders the coverage summary, the uncovered-deliverables
  callout, per-proposal "delivers" chips, and the creep flag; the accept path still calls
  `createTicket` with `parent_epic_id` (unchanged). "Would a test catch it?" is settled by breaking
  the derivation, not by argument.

**Verify / tripwire.** `npm run verify` run in full by me. The live-suite test count must show the

## 7. Review

Ordinary story → **one strong adversarial whole-branch reviewer + a security reviewer**, not the
multi-agent deep pass. R2.1 extends an existing endpoint's *response* shape; it does **not** touch the
JWT verify, add a secret, change RLS, or touch the CI gate — so it is not a security-boundary diff by
the CLAUDE.md test. The security reviewer still runs (every feature build), primarily to confirm that
non-boundary claim holds and that the auth seam is genuinely untouched.

## 8. Forward compatibility (keeps R2.2 cheap)

- The response stays additive; R2.2 (grounded estimation) plugs into the same `llm.py` seam and can
  add estimate fields per proposal without reshaping this contract.
- `traceability.py` being a pure module means richer grounding later (e.g. context spans, or
  persisting the trace at Rung 3) is a change in one isolated place, not across the endpoint.
- Persisting the trace remains a clean future step: `covers` already travels to the client, so a later
  slice can write it onto the ticket (additive column/side-table) without changing the service.
