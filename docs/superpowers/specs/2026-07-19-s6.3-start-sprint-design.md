# S6.3 — Start sprint (enforce one active) — design

## Story & acceptance criteria

From `docs/sprintboard_phase1_backlog.md`, E6 Sprints:

> ### S6.3 Start sprint (enforce one active)
> - **AC:** starting sets status active.
> - **AC:** attempting to start a second sprint while one is active is rejected by the
>   partial unique index and surfaced with a clear message.

## What already exists (verified against schema and code)

- `sprints.status` is `text not null default 'future' check (status in ('future','active','complete'))`.
- The one-active rule is **already enforced** by a partial unique index, present since the
  original schema:
  ```sql
  create unique index sprints_one_active_per_project
    on sprints(project_id) where status = 'active';
  ```
  **No migration is needed.** CLAUDE.md is explicit: "One active sprint per project: enforced by a
  partial unique index. Surface the rejection as a clear message. **Do not work around the index.**"
- `src/lib/sprints.ts` has `createSprint` (never sends status) and `listSprints`. No update path yet.
- `ProjectShell` owns the sprint list as a three-state (`loading`/`loaded`/`failed`) read and shares
  it via `ProjectShellContext`. It already has `onSprintCreated` (prepend) and `onTicketUpdated`
  (replace-by-id) reducers to mirror.
- `SprintsTab` is a pure view of that list. Each row shows name, goal, dates, ticket count, and a
  status badge.
- The `blockTicket` / `unblockTicket` functions are the precedent for an **intent-named** write that
  returns a **discriminated result** distinguishing a user-correctable outcome from `'unknown'`.

## Design

### 1. Data layer — `startSprint(id)` in `src/lib/sprints.ts`

A new intent-named function that flips one sprint to `active` and lets the database enforce the
one-active rule:

```ts
export type StartSprintResult =
  | { ok: true; sprint: Sprint }
  | { ok: false; error: 'already_active' }
  | { ok: false; error: 'unknown' }

export async function startSprint(id: string): Promise<StartSprintResult> {
  const { data, error } = await supabase
    .from('sprints')
    .update({ status: 'active' } satisfies SprintStatusUpdate)
    .eq('id', id)
    .select()
    .single()

  if (error) {
    // 23505 = unique_violation: the partial unique index rejected a SECOND active sprint.
    // This is the ONLY user-correctable failure here (complete the current one first), so it
    // gets its own tag and a clear message at the UI. Everything else — RLS zero-row match,
    // network — is not user-correctable and collapses to 'unknown', exactly like createSprint.
    if (error.code === '23505') return { ok: false, error: 'already_active' }
    return { ok: false, error: 'unknown' }
  }
  return { ok: true, sprint: data as Sprint }
}
```

**Decisions (made autonomously; rationale recorded here):**

- **Three-variant result, not `createSprint`'s single `'unknown'`.** The AC demands a *clear
  message* for the one-active rejection. That rejection is a distinct, user-correctable state
  ("finish the active sprint first"), so it earns its own tag. `createSprint` had no
  user-correctable failure, which is why it collapsed everything to `'unknown'`; this is the
  deliberate, principled difference, not an inconsistency.
- **Never deactivate another sprint to make room.** We attempt the update and let the index reject
  a second active. CLAUDE.md forbids working around the index, and doing so would also silently
  end a running sprint — a destructive side effect the user never asked for.
- **Branch on Postgres error code `23505` (unique_violation).** This is the standard code the
  partial unique index raises and PostgREST surfaces as `error.code`. The **live integration test
  proves this exact code** against the real index — the branch is not taken on faith.
- **`SprintStatusUpdate = Pick<TablesUpdate<'sprints'>, 'status'>`** added to `domain.ts`, used via
  `satisfies` at the write site. Parity with the codebase's typed writes (`TicketBlockUpdate`,
  `SprintCreateInsert`): the write's shape is structural, not a comment. S6.4 (complete sprint)
  reuses the same type — this is typing, not speculative generality.

### 2. Shell wiring — `onSprintUpdated` in `ProjectShell`

Add a generic replace-by-id reducer, mirroring `onTicketUpdated` exactly, and expose it on
`ProjectShellContext`:

```ts
const onSprintUpdated = (updated: Sprint) =>
  setSprintsLoaded((prev) =>
    prev && prev.projectId === project.id && prev.phase === 'loaded'
      ? { ...prev, sprints: prev.sprints.map((s) => (s.id === updated.id ? updated : s)) }
      : prev,
  )
```

Generic name (`onSprintUpdated`), because starting is one update among several the list will accept
(S6.4 completes). The `phase === 'loaded'` guard is load-bearing for the same reason it is on
`onTicketUpdated` — a `failed`/`loading` variant has no `sprints` to map, and reconstructing one
would resurrect the "a failed read looks successful" defect S4.6 removed. Only the started sprint
changes: enforcing one-active by *rejection* means no other row is touched, so a single replace-by-id
is complete and correct.

### 3. UI — `StartSprintButton` component + `SprintsTab` row

A focused component owning the async call, pending state, and error message for one sprint row:

- Renders a `Start` button.
- On click → `startSprint(sprint.id)`, button shows pending and is disabled.
- On `ok` → calls `onStarted(result.sprint)` (the shell's `onSprintUpdated`).
- On `already_active` → renders a clear inline message: **"This project already has an active
  sprint. Complete it before starting another."**
- On `unknown` → **"Something went wrong. Please try again."** (matches `CreateSprintDialog`).

`SprintsTab` renders `<StartSprintButton sprint={sprint} onStarted={onSprintUpdated} />` **only for
`future` sprints**. An `active` sprint is already started; a `complete` sprint is terminal
(reactivation is out of scope and has no UI path). The error is shown at the row where the user
clicked, because that is where the action originated and where the eye already is.

**Why a dedicated component, not inline state in `SprintsTab`:** it isolates the one piece of async
state and error handling into a unit that is trivially unit-testable in isolation, and keeps
`SprintsTab` a pure view (its established role). Mirrors how the create flow lives in
`CreateSprintDialog`, not inline in the tab.

## Testing

- **`src/lib/sprints.test.ts`** — unit-test `startSprint` with the mocked supabase builder
  (`from().update().eq().select().single()`): success returns the updated sprint; `{ code: '23505' }`
  → `already_active`; any other error → `unknown`.
- **`src/routes/StartSprintButton.test.tsx`** — renders a Start button; click calls `startSprint` and
  on success calls `onStarted` with the returned sprint; `already_active` shows the clear message and
  does **not** call `onStarted`; `unknown` shows the generic message; button disabled while pending.
- **`src/routes/SprintsTab.test.tsx`** — Start button renders for a `future` sprint and **not** for
  `active` or `complete`; the button is wired to `onSprintUpdated`.
- **`src/routes/ProjectShell.test.tsx`** — the `onSprintUpdated` reducer replaces the started sprint
  in the shared list by id and leaves others untouched (the seam per-task mocks cannot see).
- **`src/test/sprints.integration.test.ts`** — the live AC proof, the important one:
  1. starting a `future` sprint sets `status` to `active` (positive control), verified by a re-read.
  2. starting a **second** sprint while one is active is rejected with `error.code === '23505'` by the
     real partial unique index — proving both that the index enforces one-active and that the code
     branches on the right signal. (Paired: the first start is the positive control for the second's
     rejection.)

## Stop-and-ask review (autopilot four exceptions)

- **Schema:** no migration, constraint, or index change — the index pre-exists. Not triggered.
- **Security boundary:** an owner-scoped `update` through the existing `sprints_owner` RLS policy; no
  auth, RLS, secret, or CI-gate change. Not triggered. (A standard security review still runs at the
  end of the build.)
- **Locked scope:** the Scrum sprint lifecycle, squarely in E6. No parked/Rung-2/Rung-3 surface. Not
  triggered.
- **AC implementable as written:** yes — this is exactly what the partial unique index exists for.
  Not triggered.

All four clear, so this design is self-approved under autopilot.
