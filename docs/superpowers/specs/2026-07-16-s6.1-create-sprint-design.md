# S6.1 — Create sprint — Design

**Story:** E6 / S6.1. **Date:** 2026-07-16. **Status:** approved, pre-implementation.

## Acceptance criteria (from `docs/sprintboard_phase1_backlog.md`)
- A sprint is created with **status future**, **optional** name, goal, dates.

## Goal and showcase framing
The first sprint surface in the app. A **Sprints** tab beside Board and Backlog lists the
project's sprints; a **Create sprint** dialog creates one. This is where the `sprint_id`
machinery shipped in E5 finally gets a UI — until now every ticket's `sprint_id` has been
null because nothing could create a sprint to point at.

## Scope discipline (locked Phase 1)
This story is **exactly**: a `sprints` data layer, a create dialog, a read-only list, and a
third nav tab. Explicitly **out of scope**:
- Adding/removing tickets to a sprint — **S6.2**.
- Start sprint / the one-active-sprint index — **S6.3**.
- Complete sprint — **S6.4**.
- Edit or delete a sprint, sprint reordering, velocity, burndown — no story exists.

**No migration.** `sprints`, its RLS policy, the composite fk, and the partial unique index
all already exist in `docs/sprintboard_phase1_schema.sql`. `Sprint`, `SprintStatus`,
`SPRINT_STATUSES` and `SprintInsert` already exist in `src/lib/domain.ts`. This story adds
no schema and no domain vocabulary.

## The AC's one real tension: `name` is `not null`

The AC says "optional name", but `sprints.name` is `text not null` with **no default**. The
three ways out were: app-side auto-naming, a migration to drop `not null`, or making the
field required in the UI (which would contradict the AC rather than implement it).

**Resolved: the app auto-names.** A blank name becomes `Sprint N`, where `N` is the
project's existing sprint count + 1. The schema is untouched, and "optional" becomes true
where it matters — at the user's fingertips. `sprints` has **no unique constraint on
name**, so duplicate names are legal: a race, or a delete-then-create, can yield two
`Sprint 3`. That is cosmetic and never corrupting, which is what makes count-based
numbering acceptable here. This is emphatically **not** the ticket-key pattern — ticket
keys are identifiers and are assigned by an atomic trigger precisely because `count(*)`
is race-prone. A sprint name is a label, not an identifier.

## Resolved design decisions

1. **`status` is never sent.** The DB default `'future'` owns it. The AC is satisfied by
   the database, not by the client.

2. **Guard the insert against `status`.** `SprintInsert` is currently the raw
   `TablesInsert<'sprints'>`, so `status` is writable — `status: 'active'` in the insert
   body **would compile today** and silently bypass S6.3's one-active-sprint rule before
   S6.3 is even built. Add to `domain.ts`:

   ```ts
   export type SprintCreateInsert = Omit<SprintInsert, 'status'>
   ```

   and bind the insert with `satisfies SprintCreateInsert`, making the guarantee
   structural rather than a comment. This applies the S5.2 lesson (`TicketInsert` leaves
   `sprint_id` optional, so adding it to the insert body would compile) *before* it bites.

3. **Dates: ISO in, ISO out — UTC-pinning is structural, not a rule.**
   `<input type="date">` is a native control: it renders in the user's own locale and we
   cannot restyle it, but the value it yields is always ISO `2026-07-20`. We store that as
   `2026-07-20T00:00:00Z` (`toUtcMidnight`) and **render it as the first ten characters of
   the UTC ISO string**.

   The off-by-one this avoids is real: a `timestamptz` of midnight UTC, formatted in a
   local zone west of UTC, renders as the **previous day**. Because the chosen display
   format is ISO, there is no reformatting step at all — no `Intl`, no locale, no
   local-timezone code path to get wrong. The hazard is designed out rather than guarded
   against, and the tests do not depend on the machine's timezone.

4. **Date ordering is validated client-side only.** zod `.refine(end >= start)` when both
   are present. No DB `check` constraint: consistent with `story_points`, which is also
   client-validated only (see the open design debts). Noted as a known asymmetry against
   CLAUDE.md's "validate at both edges", not an oversight.

5. **Sprint state lives in `SprintsTab`, not `ProjectShellContext`.** Tickets are hoisted
   into the shell because Board *and* Backlog consume them; sprints have exactly one
   consumer today. S6.2 will need them in the backlog and will hoist them then.

   Deliberate: the shell's ticket loading has a known weak spot — `loadingTickets` is
   derived purely from project-id tagging and so **structurally cannot** represent
   "failed", meaning a paused database renders as "Nothing in the backlog." Hoisting
   sprints into that shape now would inherit the defect into a second feature. New code
   gets a real three-state load instead: **loading / loaded / failed**, with an actual
   error message on failure.

### Data layer
```ts
createSprint(input: {
  projectId: string
  name?: string
  goal?: string
  startDate?: string   // ISO date, 'YYYY-MM-DD'
  endDate?: string
}): Promise<{ ok: true; sprint: Sprint } | { ok: false; error: 'unknown' }>

listSprints(projectId: string): Promise<Sprint[]>
```

`sprints` has **no `owner_id`** — the `sprints_owner` RLS policy reaches through the
project, exactly like `tickets_owner`. A cross-tenant insert is rejected by the database,
not by this function. `listSprints` filters on `project_id` explicitly: RLS scopes the
select to the owner, but **an owner has many projects**, so without the filter it returns
every project's sprints. Same reasoning as `listTickets`.

`defaultSprintName` is a pure function taking the existing sprints — unit-testable with no
database.

## Testing

**Unit**
- `defaultSprintName`: blank → `Sprint 1`; whitespace-only → treated as blank; numbering
  off existing sprint count; a supplied name is trimmed and kept.
- `sprint-schemas`: every field optional; end-before-start rejected; end == start allowed;
  either date alone allowed.
- `toUtcMidnight` / `formatSprintDate`: round-trips the typed day.

**Component**
- Create with a typed name → appears in the list.
- Create with a blank name → lands `Sprint N`.
- Failed create surfaces an error and keeps the dialog open.
- Failed load renders the error state, **not** an empty list.

**Live integration (real database)**
- Create → `status` comes back `'future'` **from the DB default**, proving the AC at the
  database rather than trusting the client.
- `listSprints` does not leak another project's sprints.
- **Guard:** the insert body carries no `status` key
  (`expect(insert).toHaveBeenCalledWith(expect.not.objectContaining({ status: expect.anything() }))`),
  so a future edit that adds one goes red.
