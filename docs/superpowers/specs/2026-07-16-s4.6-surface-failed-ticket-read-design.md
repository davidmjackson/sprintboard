# S4.6 — Surface a failed ticket read, with retry

**Story:** S4.6 (E4 Tickets). **Date:** 2026-07-16.

The debt this pays off is recorded in five places in the codebase, each politely declining to fix
it: `ProjectShell.tsx`, `BacklogTab.tsx`, `SprintsTab.tsx`, and two memory entries. S6.2's
whole-branch review is what forced it — the sprint count read `0` off a failed ticket read, and the
one-line gate that fixed the *loading* lie could not touch the *failed* one, because the state
needed to tell "empty" from "broken" does not exist.

Autopilot run. Scope (error state **plus** retry) is David's explicit call. No other decision
touched the schema, a security boundary, or locked scope; the ACs are newly authored here, so none
was unimplementable.

---

## The defect, precisely

`ProjectShell.tsx`:

```ts
.catch(() => active && setLoaded({ projectId: activeProjectId, tickets: [] }))
```

A rejected `listTickets` **resolves** the load with an empty list. `loadingTickets` is derived as
`loaded?.projectId !== project.id`, so after the catch it is **`false`** — the read looks finished
and successful. "Failed" is not merely unhandled; it is **unrepresentable**.

## What each dependent lies about today

Three consumers, and the survey turned up one more lie than the story was scoped for:

| Surface | While loading | On failure |
|---|---|---|
| `BacklogTab` | "Loading…" ✓ | **"Nothing in the backlog."** ✗ |
| `BoardTab` | **"No tickets yet." ×4** ✗ | **"No tickets yet." ×4** ✗ |
| `SprintsTab` count | `—` ✓ (S6.2) | **`0 tickets`** ✗ |

**`BoardTab` never reads `loadingTickets` at all** — it was not found by the S6.2 review, only by
this story's survey. It has been claiming an empty board during every load since E7. It is in scope:
the story is "surface a failed read", and the board is the app's front page.

## Design

### 1. A tagged, three-state read — the shape S6.1 already proved

`ProjectShell` gets for tickets exactly what S6.1 gave sprints, and what S6.2 preserved through the
hoist:

```ts
type TicketsLoaded =
  | { projectId: string; nonce: number; phase: 'loaded'; tickets: Ticket[] }
  | { projectId: string; nonce: number; phase: 'failed' }
```

`loading` stays **derived** ("no result tagged for this project *and* this attempt has landed yet"),
never a synchronous `setState` at the top of the effect — `react-hooks/set-state-in-effect` is a
build-blocking lint rule here, and the derived form also prevents flashing the previous project's
data on a fast switch.

**`ticketsPhase` replaces `loadingTickets` on the context.** Not added alongside it: two fields
encoding one fact is how they drift, and `loadingTickets: false` would keep meaning both "loaded"
and "failed" for any consumer that kept reading it. Replacing it makes every consumer a compile
error until it has considered failure — which is the point. `ProjectShellContext` already carries
`sprintsPhase`, so this makes the two reads symmetrical rather than inventing a shape.

### 2. Retry via a nonce — and why the nonce is in the tag

The effect keys on `[activeProjectId]`, so nothing re-runs it. A `reloadNonce` in the deps fixes
that, and `onRetry` bumps it.

**The nonce must also be part of the loaded tag.** Without it, retrying from `failed` leaves the
stale `failed` result matching the current project, so the UI would keep showing the error until the
new result lands — a Retry button that appears to do nothing for a second, which is exactly how a
user ends up hammering it. With the nonce in the tag, the old result stops matching the moment the
nonce bumps, the phase derives back to `loading`, and the button gives immediate honest feedback.
This keeps "loading" derived, so no lint rule is bent to get there.

**One nonce drives both reads.** Retry means "reload this project's data", not "reload precisely the
thing that failed". If tickets failed and sprints are fine, the sprint refetch is harmless; the
alternative — per-read nonces and per-read retry buttons — is more machinery to explain a
distinction the user does not have. **This is what closes S6.2's sticky sprint read**: that
regression exists solely because nothing could re-run the shell's effect.

### 3. One shared failure component

`BacklogTab`, `BoardTab` and `SprintsTab` all need "we could not load this — try again". That is one
component (`LoadFailure`), not three copies: identical copy and behaviour duplicated three times is
how the third one drifts. It takes a message and `onRetry`, and owns the `role="alert"` and the
button.

`SprintsTab`'s existing sprint-failure block adopts it, replacing its "Please refresh to try again"
copy — that sentence was an admission that no in-app recovery existed, and it is now false.

### 4. The count, on a failed ticket read

The badge cannot hold a Retry button, and the sprint list around it is fine — the sprints loaded.
So the count renders `—` with `sr-only` text distinguishing *unavailable* from *loading*. Silent and
honest: it does not claim a number, and it does not manufacture a page-level error out of a
degraded badge. The Backlog and Board carry the retry affordance for the same underlying failure.

### 5. What is NOT in this story

- **No change to `listTickets`/`listSprints`.** They already reject properly; the bug was always the
  shell swallowing it. The data layer is untouched.
- **No automatic retry, no backoff, no polling.** A manual Retry is what the ACs ask for. Automatic
  retry against a paused Supabase project is a way to look busy while achieving nothing.
- **No error taxonomy.** "Could not load" covers paused DB, network, and RLS alike; distinguishing
  them needs error shapes `listTickets` does not return, and Phase 1 has no story for it.
- **The `commit()` `mountedRef` rollback gap** (a write failing after the dialog closes) is a
  different debt — a *write* path, not a read. Not touched.

## Testing

Every AC is client-side, so the unit suites carry them; **no new live test** — this story adds no
database behaviour, and `listTickets` already rejecting is what the shell now honours.

- `ProjectShell.test.tsx`: a rejected `listTickets` puts `ticketsPhase: 'failed'` on the context; a
  rejected `listSprints` still yields `sprintsPhase: 'failed'`; **retry re-runs both reads** and a
  read that fails then succeeds renders the data (the recovery path, end to end, through the real
  tabs); retry from `failed` returns to `loading` first.
- `BacklogTab` / `BoardTab`: failed → error + Retry, not the empty state; **empty → the empty state,
  not an error** (the positive control that keeps AC 2 honest — without it, a component that always
  errored would pass).
- `SprintsTab`: failed ticket read → `—`, never `0`; count still correct when loaded.
- **Every one of these is proven by breaking the code**, per the project's standing rule: a passing
  test proves nothing until it has been watched to fail. The `sprints.length === 0` incident on S6.2
  is the precedent — five test cases all stayed green against the wrong guard.
