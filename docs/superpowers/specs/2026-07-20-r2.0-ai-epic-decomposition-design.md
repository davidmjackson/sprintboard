# R2.0 — Walking skeleton: AI epic decomposition (design)

**Date:** 2026-07-20
**Status:** approved (brainstorming) — pending written-spec review, then implementation plan
**Rung:** 2 (first slice). This story deliberately opens the parked Rung 2 door.
**Branch:** `r2.0-ai-epic-decomposition`

---

## 1. Context

Phase 1 (Rung 1) is complete: a working owner-scoped Scrum board, React → Supabase,
no backend. The project charter states the point of the whole exercise:

> a credible Jira-style board with a grounded-AI layer added at Rung 2. A clean working
> slice plus defensible AI.

The board now exists to carry that AI layer. This story is the **first slice of Rung 2** —
the headline feature (AI epic decomposition) in its simplest honest form, built as a thin
vertical slice that proves the entire new path end to end before we thicken it.

The schema already anticipates this: `tickets.context` (text) and `tickets.deliverables`
(jsonb) exist on epics, commented *"Feed the Rung 2 AI decomposition feature,"* and children
link to their epic via `parent_epic_id` (composite fk). **No schema change is required.**

## 2. Goal (this slice)

Given an epic's `context` + `deliverables`, let a user ask Claude to propose child stories,
review the proposals, and accept some — which then become real child tickets on the board,
linked to the epic. Prove the whole path — **React → a new local FastAPI service → Claude →
back → persist via existing code** — is real and correct.

## 3. Standing decisions (made during brainstorming, binding)

Each was an explicit user decision; recorded here so the plan doesn't relitigate them.

| Decision | Choice | Why |
|---|---|---|
| **Fidelity** | Real, **local-only** | The FastAPI + Claude integration genuinely works when run locally (real API calls on demand). No cloud deploy — proves the engineering without hosting cost/ops. CI mocks the model. |
| **Data relationship** | **Stateless service** | The service verifies the JWT (authN only), calls Claude, returns proposals. It touches **no** database — no service-role key, no RLS re-implementation on a brand-new tier. All persistence stays client-side under existing RLS. Tiniest security surface. |
| **Slice boundary** | **Thin vertical slice** | Service + auth seam + one real (minimal) decomposition endpoint + a minimal React affordance, proven end to end. Richer decomposition (traceability, gap/creep detection) comes later. |
| **Loop closure** | **Display + accept** | Proposals are shown; the user accepts all/a subset; React creates them as real child tickets via the existing tested create path. Reuses solved code, so a genuinely usable feature at small marginal cost. Inline editing of proposals before accept is deferred. |
| **Model** | `claude-opus-4-8` | Correct default per the Anthropic guidance; structured-output generation task. Swappable via env if a cheaper tier is ever wanted — not a code change to the contract. |

## 4. Architecture

A new **Python/FastAPI service in an `api/` subdirectory** of this same repo (monorepo — one
repo per suite app), alongside the existing React `src/`. Runs locally via `uvicorn`; no deploy.

```
Browser (React)                    api/  (FastAPI, local)              Anthropic
──────────────                     ────────────────────               ─────────
Epic detail dialog                 POST /decompose
  "Decompose with AI"  ──JWT+body──▶  verify Supabase JWT (authN)
                                      build grounded prompt  ──────────▶ Claude
  proposals panel      ◀──JSON─────  return proposed stories ◀────────── (structured out)
  accept (all/subset)
      │
      └─▶ existing tickets.ts create path ──▶ Supabase (RLS, parent_epic_id)
```

The single click's journey: React already holds the epic's `context`/`deliverables` (it
rendered them), so it **sends** them rather than making the service fetch them. The service
authenticates and calls Claude. On accept, the **browser** persists — through code that
already exists and is already tested.

### Proposed layout

```
api/
  app/
    main.py            # FastAPI app, CORS (localhost dev origin), route registration
    auth.py            # Supabase-JWT verification dependency (authN)
    decompose.py       # POST /decompose handler
    schemas.py         # pydantic request/response models
    llm.py             # thin Claude wrapper — the ONE mockable seam
  tests/
    test_auth.py       # valid JWT passes; missing/expired/garbage → 401
    test_decompose.py  # stubbed proposals flow through; bad bodies rejected
  pyproject.toml       # fastapi, uvicorn, anthropic, pyjwt, pydantic + ruff + pytest
```

## 5. The auth seam (security boundary)

React attaches the current Supabase JWT as `Authorization: Bearer <token>`. A FastAPI
dependency **verifies that token locally** — signature, `exp`, and audience — and rejects
anything unsigned/expired/garbage with **401**. It reads the user id from the verified claims;
**no network call** to Supabase (avoids the auth-rate-limit class of problem entirely, and
keeps the endpoint fast).

The service authenticates but does **not** authorize against data — it never reads a row, so
there is nothing to owner-scope on the server. Authorization stays where it already lives: in
Supabase RLS, enforced when the browser persists accepted tickets.

**Open question for the plan:** Supabase JWTs are signed either with an HS256 shared secret
(legacy) or asymmetric keys via JWKS (newer projects). Which this project uses must be
confirmed against the actual token/project settings before implementing `auth.py`, because it
changes which secret/material the service holds. Both yield the same security property (reject
anything not signed by Supabase; check `exp`/`aud`).

## 6. The decomposition endpoint

`POST /decompose`, `Authorization: Bearer <supabase-jwt>`.

```jsonc
// Request
{ "epic": { "summary": "string", "context": "string", "deliverables": ["string", ...] } }

// Response
{ "proposals": [
    { "title": "string", "description": "string", "type": "story|bug|task",
      "rationale": "which deliverable/context this serves" }
] }
```

- `type` is constrained to `story | bug | task` (never `epic` — decomposition produces
  children). `rationale` is **display-only** in this slice (a forward-nod to R2.1's real
  traceability), not persisted.
- Contracts are pydantic models in `schemas.py`; validation errors return **422**.

### The Claude call (`llm.py` — the one mockable seam)

- **Model `claude-opus-4-8`**, adaptive thinking on, `max_tokens ≈ 4096` (small output →
  non-streaming is fine).
- **Structured output** via `output_config={"format": {"type": "json_schema", "schema":
  PROPOSALS_SCHEMA}}` so the reply is guaranteed-parseable JSON matching the response contract
  — no brittle text scraping. The epic's `context` + `deliverables` go in the user turn; a
  system prompt frames the Scrum decomposition task (propose child stories that build the epic,
  one per deliverable where sensible, non-epic types only).
  - JSON-schema-limitation note (per Anthropic structured-output rules): every object needs
    `additionalProperties: false` + `required`; no `minLength`/`maxLength`/numeric constraints.
    The schema stays flat and simple accordingly.
- `llm.py` exposes one function (roughly `propose(epic) -> Proposals`). **This is the single
  seam CI patches**: tests inject canned proposals, so CI needs no `ANTHROPIC_API_KEY` and makes
  zero network calls. Locally, the real key drives a real call.

## 7. React integration

- A **"Decompose with AI"** button in the existing epic detail dialog, shown only when
  `type === 'epic'`.
- It POSTs the epic body to the service (base URL from a **public** `VITE_AI_API_URL`, default
  `http://localhost:8787` — a URL, not a secret) with the current Supabase JWT attached.
- Proposals render in a panel with per-item checkboxes (accept all or a subset).
- **On accept**, for each selected proposal React calls the existing, tested `createTicket`
  (`src/lib/tickets.ts`) with `parent_epic_id` = the epic, `status = 'To Do'`, `sprint_id =
  null` — so children land in the backlog and on the board through the exact path every other
  ticket uses. No new write path, no new RLS surface.

The whole loop: **epic dialog → local FastAPI (verifies JWT) → Claude (structured JSON) →
proposals panel → accept → existing `createTicket` under RLS.**

## 8. Secrets & config

- **`ANTHROPIC_API_KEY`** and the Supabase JWT verification material: **server-side only** —
  from env, **never** `VITE_`-prefixed, never in the browser bundle. Mirrors exactly how
  `SUPABASE_SERVICE_ROLE_KEY` is handled today. `check-bundle.mjs` continues to guarantee no
  privileged key reaches `dist/`.
- **`VITE_AI_API_URL`** is public (a localhost URL) and browser-side — this is fine; it is not
  a secret.
- Local env lives in `.env.local` (already gitignored); the plan will document exactly which
  keys the service reads.

## 9. Testing & CI

- **Python (`api/tests/`):** pytest, model stubbed via `llm.py`. Real coverage of the **auth
  seam** (valid JWT passes; missing/expired/garbage → 401) and the **endpoint** (stubbed
  proposals flow request → response; bad bodies → 422). No key, no network, deterministic.
- **New CI job `api.yml`:** `ruff` (lint/format) + `pytest`. **Separate from `verify`** — the
  existing gate workflow is byte-untouched. The Python job may be a required check in its own
  right, but stays its own workflow so the TS gate and Python gate never entangle. (Rationale
  lives in CLAUDE.md: `verify` is *the* gate and must not be widened or weakened.)
- **React:** Vitest coverage of the button + proposals panel + accept flow, with `fetch`
  mocked (jsdom has no network) — asserting the accept path calls `createTicket` with
  `parent_epic_id` set. This moves the Vitest counts; new numbers recorded in the status block.
- **Tripwire:** Python tests do not move the Vitest count (418/358). The Python test count is
  recorded separately as a new baseline.
- **No E2E, no live-model test in CI.** Real Claude calls happen only when run locally
  (`uvicorn` + a real key). This is the "real, local-only" decision holding.

## 10. Explicitly deferred (YAGNI — not in this slice)

- **Traceability spans, coverage-gap & scope-creep detection** → R2.1. Here `rationale` is
  display-only.
- **Grounded estimation** → R2.2.
- **Editing a proposal before accepting** → later; this slice accepts proposals as-is.
- **Cloud deployment of the service** → parked by the local-only decision.
- **Streaming the model response, retry/rate-limit UX polish, response caching** → not needed
  at this scale.
- **The service reading/writing the DB** → ruled out by the stateless decision.

## 11. Forward compatibility (keeps R2.1/R2.2 cheap)

- The stateless request/response shape extends naturally: R2.1's traceability makes `rationale`
  a real structured link to a specific deliverable/context span (persisted), and adds
  gap/creep analysis as extra response fields — additive, no rework of the tier.
- Keeping persistence in `createTicket` means the AI path never diverges from the hand-created
  path; new ticket fields land in one place.
- The `llm.py` seam is where estimation (R2.2) and richer prompts plug in.

## 12. Open questions to resolve in the plan / at implementation

1. **Supabase JWT signing scheme** (HS256 shared-secret vs JWKS) — confirm against the real
   project before writing `auth.py` (§5).
2. **Service port** — proposal `8787`; confirm no conflict with the Vite dev server.
3. **Exact env var names** for the service (`ANTHROPIC_API_KEY`, Supabase JWT material) — pin
   in the plan and document in `.env.local` + CLAUDE.md.
4. Whether the new `api.yml` check is marked **required** on the branch ruleset (leaning yes —
   deterministic and fast) — a GitHub server-side setting, decided at ship time.
