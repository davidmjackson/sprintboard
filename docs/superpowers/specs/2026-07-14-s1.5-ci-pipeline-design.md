# S1.5 — CI pipeline (SPRIN-13)

**Date:** 2026-07-14
**Branch:** `feat/s1.5-ci`
**Story:** S1.5 CI pipeline

## Acceptance criteria

1. Every PR runs lint, typecheck, and tests.
2. A failing test blocks merge.

AC2 is the one with teeth. A workflow that runs on a PR satisfies AC1 on its own;
nothing about a green run establishes AC2. See "Proving the gate bites" below.

## Why this story exists

Three stories (S1.1, S1.2, S1.3) reached review with no gate on any PR. Last session a
branch that did not compile was pushed and PR'd, and the only thing that caught it was a
person choosing to look. Every check on this project so far has been one an agent or a
human elected to run, assembled by hand, from memory.

That is a design problem, not a discipline problem. There is no single command that
means "is this green". The gates exist as four separate npm scripts, so the check list
has to be reassembled correctly on every occasion — and once, it wasn't: `tsc --noEmit`
was used as a stand-in for `npm run build`, passed, and hid a red build.

CLAUDE.md already states the principle this story applies:

> Guard hooks are active and authoritative. Prompt directives are requests; hooks are
> enforcement.

A written rule saying "always run the checks" would be one more request. This story
builds the enforcement.

## Design

### 1. One command: `npm run verify`

```json
"verify": "npm run lint && npm run format:check && npm run build && npm test"
```

Ordered cheapest-first so it fails fast. `build` is already
`tsc -b && vite build && node scripts/check-bundle.mjs`, so the typecheck in AC1 and the
bundle credential guard are both inside it.

This is the only blessed answer to "is it green". CI runs this exact command and nothing
else, so the set of gates cannot drift between a developer's machine and the pipeline:
there is one definition, in `package.json`, and both sides call it.

### 2. Pin the Node version: `.nvmrc`

Nothing currently records which Node this project builds on. That is the same class of
drift, so: `.nvmrc` pinning Node 20 (matching the development environment), and CI reads
the version from it rather than hardcoding a second copy.

### 3. `.github/workflows/verify.yml`

One job, `verify`, triggered on `pull_request` targeting `main` and on `push` to `main`.

Steps: checkout → `setup-node` (version from `.nvmrc`, npm cache enabled) → `npm ci` →
`npm run verify`. Permissions are `contents: read` only.

The job name `verify` is the string the branch ruleset requires.

**A required check that never reports fails closed.** Verified empirically on this
repository (PR #7, ruleset live, no workflow yet, zero checks ever run):

```
mergeable       = true      # only means "no merge conflicts" — not "you may merge"
mergeable_state = blocked   # the merge is refused
```

So renaming the job, deleting the workflow, or filtering it out of a run **freezes
merges loudly** rather than waving them through. That is the safe direction, and it means
the job name is not the fragile part of this design.

**The fail-open case is a job that reports success without running.** Three ways to
create one, all of which look innocent in a diff:

1. **An `if:` condition on the job.** A skipped job's check run concludes as `skipped`,
   which GitHub accepts as satisfying a required check. The gate goes green having
   verified nothing. This is the one that would actually hurt.
2. **A `strategy.matrix`.** The check is then named `verify (20)`, and the required
   context `verify` never matches.
3. **A `paths:` / `paths-ignore:` filter** on the trigger.

Therefore the verify job carries **no `if:`, no matrix, and no paths filter**, and the
workflow says so in a comment. These are the three edits a future contributor would make
without suspecting they were disarming the gate.

Note in passing: `mergeable: true` above is exactly the sort of confidently-wrong reading
the Verification section warns about. It is not the field that decides anything;
`mergeable_state` is.

**No `cancel-in-progress`.** The RLS suite creates real rows in the shared Supabase
database and removes them in `afterAll`. Cancelling a run mid-suite strands a project,
its sprint, its tickets and its counter row in the live database, permanently, on every
cancelled run. A wasted minute of CI is far cheaper than leaked fixtures, so superseded
runs are allowed to finish.

### 4. Credentials: two secrets, four variables

The RLS suite needs six values. Only two of them are secret.

| Value | Where | Why |
|---|---|---|
| `RLS_TEST_A_PASSWORD` | Actions **secret** | A live password. |
| `RLS_TEST_B_PASSWORD` | Actions **secret** | A live password. |
| `VITE_SUPABASE_URL` | Actions **variable** | Ships to every browser in `dist/`. |
| `VITE_SUPABASE_ANON_KEY` | Actions **variable** | Ships to every browser in `dist/`. |
| `RLS_TEST_A_EMAIL` | Actions **variable** | Already committed in `.env.example`. |
| `RLS_TEST_B_EMAIL` | Actions **variable** | Already committed in `.env.example`. |

`.env.example` states the URL and anon key are safe to expose — the security boundary is
owner-scoped RLS, not secrecy of those keys. Storing them as secrets would imply the
opposite and quietly contradict the project's own threat model. The split says what is
true.

The values are set with `gh secret set` / `gh variable set`, which prompt for the value
interactively. No password is ever written into a command line, a shell history, a file,
or an agent's context.

### 5. Ruleset on `main`

- Require a pull request before merging, with **0 required approvals**. GitHub does not
  permit self-approval; on a solo repository, requiring one review would deadlock every
  PR permanently.
- Require the `verify` status check to pass.
- Block force-pushes and branch deletion.
- **No bypass actors.** An admin override makes the gate advisory again, which is the
  failure mode this story exists to close. Overriding should require deliberately editing
  the ruleset — a visible, auditable act, not a moment of haste.

### 6. CLAUDE.md

> Two "green" checks have been reported on this project that were not green. Both had the
> same shape: the check that ran was not the check that was claimed.
>
> - **Verification means `npm run verify`.** Never a hand-assembled subset, never a proxy.
>   `tsc --noEmit` is not `npm run build` — it passed on a branch whose build was red. CI
>   runs this same command, so local and CI cannot drift.
> - **Compare against `origin/*`, and fetch first.** A stale local `main` made a correct

> - **A surprising result is a hypothesis, not a finding.** Before acting on or reporting
>   something alarming, re-derive it a second, independent way.

Reconcile the existing CI paragraph in the same edit. It currently reads "CI must run
`npm test`, never `npm run test:unit`". Once `verify` exists that wording is stale and
competes with the new rule. It becomes: CI runs `npm run verify`, which includes
`npm test`; `test:unit` stays a local fast-loop convenience and must never be the gate,
because it excludes the RLS suite and would leave "RLS still holds" unmet on every PR.

## Proving the gate bites

A green CI run proves AC1. It proves nothing whatsoever about AC2 — a working gate and a
purely decorative one both display a green tick. This is the same trap as an RLS test
that asserts `error === null`: it passes just as happily against a wide-open database.

So AC2 is established by **counterfactual**, exactly as the RLS suite was:

1. With the workflow and ruleset live, push a commit to the story branch containing a
   deliberately failing test.
2. Confirm the `verify` check reports failure, and that GitHub reports the PR as **not
   mergeable** — the merge button blocked, not merely a red tick beside an enabled one.
3. Record that result, then revert the commit and confirm the PR returns to mergeable.

Without step 2 we would have built a gate we merely believe in, which is precisely the
state the project is in today.

A second, cheaper counterfactual covers the credential wiring: the RLS suite already
throws rather than skips when `CI` is set and credentials are missing
(`assertCredentialsOrExplain`). If the secrets are misconfigured, CI goes red rather than
green-with-13-tests-quietly-skipped. That behaviour is inherited from S1.3 and needs no
new code, but the first CI run must be checked to confirm 43 tests ran, not 30.

## Accepted limits

- **Fork PRs receive no secrets**, so the RLS suite hard-fails for an outside
  contributor. This is correct behaviour rather than a bug — refusing to skip a security
  test is the point — and is acceptable for a solo portfolio repository.
- **A paused Supabase project now blocks merges.** The free tier pauses on inactivity, so
  wiring the live RLS suite into a required check makes S1.4's keepalive load-bearing
  rather than a nicety. S1.4 should follow this story.

## Out of scope

Deployment, preview environments, coverage reporting, a test matrix across Node versions,
and caching beyond `setup-node`'s npm cache. This story builds a gate, not a pipeline.
