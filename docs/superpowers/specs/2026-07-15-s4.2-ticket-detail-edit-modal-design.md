# S4.2 — Ticket detail and edit modal — Design

**Story:** E4 / S4.2. **Date:** 2026-07-15. **Status:** approved, pre-implementation.

## Acceptance criteria (from `docs/sprintboard_phase1_backlog.md`)
- Opening a ticket shows all fields and allows editing summary, description, type,
  assignee, points, acceptance criteria, labels.
- Saving updates the record and refreshes `updated_at`.

## Goal and showcase framing
A Jira-style ticket detail modal with **inline, click-to-edit** fields. The Jira look is
deliberate: this is a portfolio/interview demo where the app must *read as Jira at a
glance*. The Jira styling is a rebrandable skin over Sprintboard's own tight Scrum model —
never shipped as a Jira competitor. Fidelity of the *interaction* (click a field → it
becomes the right editor in place) is the point of this story.

## Scope discipline (locked Phase 1)
This story is **exactly** the seven AC fields + inline editing + the two-column Jira skin.
Explicitly **out of scope**, each its own later story/PR:
- Status change (board drag) — later.
- Block / unblock — **S4.4**.
- Epic `context` + `deliverables` — **S4.5**.
- Sprints — E6.

Status and any blocked marker appear **read-only** in this modal; the block/unblock and
epic editors are not built here. No parked (Rung 2/3) feature is introduced.

## Two resolved design decisions
1. **Assignee = Jira-shaped inline dropdown, contents `{ Unassigned, You }`.** Phase 1 is
   single-owner RLS (`profiles_self`): the only user anyone can see is themselves; teams are
   Rung 3. The *control* looks like Jira's assignee dropdown (click → dropdown of names);
   the *list* holds one real name now. **Do not widen the `profiles` SELECT policy** — the
   `display_name` fallback is the user's email, so a multi-user picker would leak every
   user's email. Read only your own `profiles` row via the existing policy for the "You"
   label. At Rung 3 the identical control fills from a membership-scoped read model.
2. **Description = basic inline plain-text editor now.** Click Description → a styled
   multiline textarea appears in place with the content; saves as plain `text`. A popular
   rich-text editor is a **later-rung seam**, not built now. No Markdown storage or render
   path yet (keeps `description` as plain `text` — no schema reshaping).

## Verified mechanics
- **`updated_at` is DB-maintained.** Trigger `tickets_set_updated_at` (BEFORE UPDATE) runs
  `set_updated_at()` → `new.updated_at := now()`. The app **must not** send `updated_at`
  (it is overwritten regardless). The AC "refreshes `updated_at`" is satisfied by the
  database; the test asserts the returned timestamp advanced.
- **RLS `tickets_owner` is `FOR ALL`** with both `using` and `with check` scoped through the
  owned project. UPDATE is covered: a cross-tenant update matches **zero rows** (RLS
  filters, it does not raise), and `with check` blocks reparenting a ticket into a project
  you do not own. This is the **first UPDATE path** on a tenant-isolated table.

## Layout — Jira two-column issue view
`shadcn` `Dialog`, opened by clicking a `TicketCard` (board) or a backlog row.
- **Header:** ticket key · type · status (read-only).
- **Left column (main):** Summary as an editable title; Description below.
- **Right sidebar (metadata):** Assignee, Labels, Story points, Type; read-only
  created/updated timestamps.
- Styling via the `frontend-design` skill at implementation time (demo-grade, not
  generic-shadcn). Inline click-to-edit applies to every editable field.

## Fields and inline editors
| Field | Editor | Notes |
|---|---|---|
| Summary | single-line text | Enter/blur commits, Esc cancels |
| Description | multiline textarea (basic/placeholder) | plain `text`; rich-text is a later seam |
| Type | native `<select>` dropdown | Epic/Story/Bug/Task from `domain.ts` |
| Assignee | dropdown | `{ Unassigned, You }`; Jira-shaped, one name now |
| Story points | number input | nullable |
| Acceptance criteria | multiline textarea | plain text |
| Labels | text input → chips | comma-separated, basic |
| key · type · status · blocked marker · created/updated | read-only | not built here |

Native `<select>` (not radix Select) per the project's jsdom convention
([[sprintboard-frontend-conventions]]).

## Interaction and save semantics
- Each field commits **independently** (Jira saves per field). A commit calls
  `updateTicket(id, { <field> })`.
- **Optimistic with rollback:** apply the new value immediately (no spinner flashes — the
  demo must feel instant); on error, revert the field to its prior value and surface a
  clear message. Each successful commit bumps `updated_at` via the trigger.

## Data layer — `src/lib/tickets.ts`
- Add `updateTicket(id, patch): Promise<UpdateTicketResult>`:
  `supabase.from('tickets').update(patch).eq('id', id).select().single()`.
- A `TicketUpdate` guard type (mirrors `TicketInsert`) — `patch` is bound
  `satisfies TicketUpdate`, forbidding immutable/DB-owned fields (`key`, `number`,
  `updated_at`, `project_id`) at compile time, so a future edit that adds one fails at the
  call site.
- Result is `{ ok: true; ticket } | { ok: false; error: 'unknown' }` (no user-correctable
  per-field constraint reachable here), consistent with `createTicket`.

## State flow — `ProjectShellContext`
- Extend the context with `onTicketUpdated(ticket: Ticket)` that **replaces** the matching
  row in `loaded.tickets` (by `id`). **No refetch** — mirrors S4.1's optimistic-append
  pattern and deliberately avoids the stale-refetch race the S4.1 deep review caught
  ([[sprintboard-review-agents-find-real-defects]]).
- The board and backlog tabs already read tickets from context, so an inline save reflects
  in both surfaces immediately.

## Testing
- **Unit:** `updateTicket` (patch shape, `updated_at` never sent); inline editors
  (click→edit→commit/cancel, Esc, optimistic rollback on error) in jsdom.
- **Live integration (`src/test/tickets.integration.test.ts`):**
  - An update persists and the returned `updated_at` **advanced** (positive control).
  - A **cross-tenant** update affects **zero rows**, and the target row is
    **independently re-read** and confirmed **unchanged** (negative + independent control —
    RLS filters, it does not raise: [[rls-tests-pass-for-the-wrong-reason]]).
- Test count moves up accordingly; update the tripwire in memory after the count is known.

## Review plan
First UPDATE path on a tenant-isolated table → standard peer + security review **plus a
deep multi-agent review**: the UPDATE `using`/`with check` is a security boundary, and the
project's rule is to run the deep pass for the first isolated write of a *verb*, not only
for machinery changes. Read the KILLED findings, not just survivors.

## Definition of Done
ACs met and covered by tests; lint + types clean; `npm run verify` green in CI (full count,
not `test:unit`); RLS two-user isolation still green; one squash-merged PR; SPRIN-23 →
Done only after merge.
