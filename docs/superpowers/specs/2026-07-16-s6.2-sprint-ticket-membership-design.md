# S6.2 — Add and remove tickets to and from a sprint

**Story:** S6.2 (E6 Sprints). **Date:** 2026-07-16.

> **AC 1:** a backlog ticket can be added to a sprint and removed back to backlog.
> **AC 2:** `sprint_id` updates correctly in both directions.

Autopilot run. Decisions below were taken without asking, per the standing decisions; none
touched the schema, a security boundary, locked scope, or an AC that cannot be implemented
as written. Each is recorded here to be vetoed.

---

## What already exists (and why this story is small)

S6.2 needs **no migration**. Checked against `docs/sprintboard_phase1_schema.sql` before
designing — the S6.1 lesson was that an AC can contradict the schema ("optional name" vs
`name not null`), so this was the first thing verified. Nothing contradicts here:

- `tickets.sprint_id` is **nullable**; the column comment already says `null = backlog`.
- `TicketUpdate` (`domain.ts`) **already permits `sprint_id`** — it is excluded only from
  the key/number/blocked sets. `updateTicket(id, { sprint_id })` compiles today.
- `tickets_sprint_fk` is a **composite** fk `(sprint_id, project_id) → sprints (id,
  project_id)`. Putting a ticket in another project's sprint is *unrepresentable*, not
  merely discouraged. Under `MATCH SIMPLE` a null `sprint_id` skips the check, so
  "removed back to backlog" passes the fk by construction. **AC 2's "both directions" is
  a schema property; this story's job is to prove it through the app's write path.**
- The **parent-epic picker** in `TicketDetailDialog` is already the exact control this
  story needs: a native `<select>`, `<option value="">No epic</option>`, committing
  `{ parent_epic_id: e.target.value || null }`. A sprint picker is that shape with `''`
  meaning backlog.
- `SprintsTab.tsx:15` predicted this story: sprints load in the tab because they had one
  consumer, and *"S6.2 will need sprints in the backlog and will hoist them then (YAGNI)."*

## Coverage gap this story must close

`tickets.integration.test.ts` already sets `sprint_id` **via the raw Supabase client**, as
S5.1 backlog fixtures. It proves the column, the composite fk and `on delete set null`. It
proves **nothing about the app's write path**. AC 2 is therefore not covered today, despite
`sprint_id` appearing throughout that suite — a green suite that looks like coverage and
is not. The new live test drives `updateTicket` in both directions.

---

## Design

### 1. The write: `commit({ sprint_id })`, no new data-layer function

Sprint membership rides the existing `commit()` in `TicketDetailDialog`, exactly as the
epic picker does. **No `addTicketToSprint`/`removeTicketFromSprint` is added.**

Considered and rejected: intent-named functions mirroring `blockTicket`/`unblockTicket`.
Those exist because the three blocked fields move together under a DB invariant
(`tickets_blocked_coherent`) that a free-form edit must never half-apply, and because
blocking carries an app-layer rule (a reason is required). **`sprint_id` has neither**: it
is one nullable column, no trigger, no multi-field invariant, and the only integrity rule
(same project) is enforced by the composite fk. An intent-named function would also have to
re-implement `commit()`'s optimistic write, rollback-on-failure and field-scoped reconcile,
or bypass them. Riding `commit()` inherits all of it for free.

`sprint_id` stays in `TicketUpdate`. Narrowing it out (the `SprintCreateInsert` move) would
be wrong here: there is no future rule being routed around, unlike `status` on a sprint
insert, which S6.3's partial unique index will own.

### 2. The read: hoist sprints into `ProjectShellContext`, keeping the 3-state

The dialog is rendered by `ProjectShell`, so the picker's options must live there.
`SprintsTab`'s own comment already booked this hoist. Context gains:

```
sprints: Sprint[]
sprintsPhase: 'loading' | 'loaded' | 'failed'
onSprintCreated: (sprint: Sprint) => void
```

**The hoist must not launder away `SprintsTab`'s three-state read.** The shell's *ticket*
read swallows a rejection into an empty list and derives `loadingTickets` purely from
project-id tagging, so it structurally cannot say "failed" — which is why a paused database
renders as "Nothing in the backlog." S6.1 deliberately did not inherit that. Moving sprints
into the same file is precisely how that defect would spread by osmosis, so the hoisted
sprint state keeps its own tagged `phase` discriminant and its `.catch()` records `failed`
rather than `[]`.

**The shell's ticket read is NOT fixed here.** It is a known, app-wide debt and its own
story; fixing it under S6.2 would widen the diff and mix two changes in one review.

`SprintsTab` becomes presentational: it reads `sprints`/`sprintsPhase`/`onSprintCreated`
from context and keeps its loading, failed and empty states verbatim. The create trigger
still renders only when `phase === 'loaded'`, for S6.1's reason (`defaultSprintName` would
otherwise number off an empty array).

### 3. The picker

In `TicketDetailDialog`, beside the parent-epic field, sourced from a new `sprints` prop
(passed by the shell exactly as `epics` already is):

- `<option value="">Backlog</option>` — selecting it is the "remove" direction. Labelled
  **Backlog**, not "No sprint": `backlog.ts` defines the backlog as `sprint_id is null`, so
  these are the same fact and the UI should use the domain's word for it.
- One option per sprint, `{name}` + its status label from `SPRINT_STATUS_LABELS`.
- **Sprints are not filtered by status.** All three are offered. Barring `complete` (or
  `active`) would be a rule the AC does not ask for, and S6.4 owns what completing a sprint
  does to its tickets. Inventing a membership rule here would pre-empt S6.3/S6.4.
- The epic picker's unavailable-option guard is mirrored (`Current sprint (unavailable)`)
  so the `<select>` stays controlled and a link is never silently dropped.
- **Disabled unless `sprintsPhase === 'loaded'`.** If the sprint read failed, an enabled
  picker offering only "Backlog" would misread as "this ticket is in no sprint" and one
  click would quietly unsprint it. Disabled is honest.

### 4. Sprint ticket counts (the discretionary call)

`SprintsTab` rows gain a ticket count, via a new `selectSprintTickets(tickets, sprintId)`
in `backlog.ts` — the natural home, beside `isBacklogTicket`, and the same rule read from
the other side. Counts are derived from the shell's already-loaded ticket list; no query.

This is the one addition not literally demanded by an AC, and the one to veto if any is.
Reasoning: AC 1 says a ticket "can be added to a sprint". Without a count, the *only*
observable effect of adding is that the ticket **disappears from the backlog** — the
removal direction is visible and the addition direction is an inference. A sprint list that
can absorb tickets but never shows them is not a working slice, and seeing the sprint's
actual tickets is S7.1's job (the board renders the active sprint), not this story's. A
count is the smallest honest thing that makes AC 1 observable in the direction it is stated.

### 5. Testing

The ACs are behavioural, so the live suite carries them; the rest is unit.

- **`tickets.integration.test.ts` (live, AC 2):** through `updateTicket` — add a backlog
  ticket to a sprint, re-read, assert `sprint_id`; set it back to null, re-read, assert
  null and that it is `is('sprint_id', null)` again. Drives the **app's** write path, which
  the existing raw-client fixtures never did.
- **`backlog.test.ts`:** `selectSprintTickets` — matches by sprint, excludes backlog and
  other sprints' tickets.
- **`TicketDetailDialog.test.tsx`:** options render; choosing a sprint commits
  `{ sprint_id: id }`; choosing Backlog commits `{ sprint_id: null }`; disabled when not
  loaded; unavailable-guard keeps the select controlled.
- **`ProjectShell.test.tsx`:** sprints load into context and reach the dialog; a failed
  sprint read surfaces `SprintsTab`'s error state, not an empty list.
- **`SprintsTab.test.tsx`:** counts render; existing loading/failed/empty states survive
  the hoist.

**"Would a test catch this?" is settled by breaking the code, not by arguing** — the S6.1
seam defect (a task mocked the dialog with a stub that never invoked the callback, so the
wiring ran nowhere) is the standing reason. The hoist creates exactly such a seam between
shell and tab, so the shell/tab wiring is proven with the real `SprintsTab`, not a stub.

## Not in this story

Starting/completing sprints (S6.3/S6.4), the board's sprint view (S7.1), fixing the shell's
ticket-read error state (app-wide debt), drag-and-drop, bulk add.
