# S1.3 — RLS two-user smoke test

**Story:** SPRIN-11. **Date:** 2026-07-13.

## Acceptance criteria

- User A cannot read, update, or delete user B's projects, sprints, or tickets.
- The check is an automated test, not a manual click-through.

## The point

CLAUDE.md's Definition of Done says *"RLS still holds (two-user isolation test
green)"* on every story from here on. This story builds the thing that sentence
refers to. It is the regression net under every future change to a policy, a
schema, or a query.

It also discharges a debt from S1.2. The trigger functions are **catalog-verified,
not behaviour-verified**: `auth.users` has zero rows, so `handle_new_user` has
never fired, `assign_ticket_key` has never assigned a key, and `freeze_ticket_key`
has never frozen one. This test is the first thing to actually run them.

## Decision: test against the live Supabase project

The alternative was a hermetic local Postgres (PGlite) with `auth.uid()` stubbed:
fast, no secrets, trivially CI-friendly. **Rejected.** It tests a *copy* of the
schema. If a future migration forgets `enable row level security`, the local test
stays green while production is wide open — which is precisely the failure this
test exists to catch. The AC is a claim about the live security boundary, so the
test must interrogate the live boundary.

The cost is accepted deliberately: network, two sets of credentials, and rows left
in the database. Mitigations below.

## Test accounts

Email confirmation is **on** for this project (the correct production setting), so
`signUp()` returns no session and the test cannot create users on the fly.

Two accounts are therefore **pre-created once**, by hand, in the dashboard with
*Auto Confirm User* ticked. The test only ever signs in. This is better than
signing up per run anyway: no unbounded accumulation of auth users that the anon
key has no permission to delete.

Credentials live in `.env.local`, and later in CI secrets.

### They must NOT be `VITE_`-prefixed

Vite inlines every `VITE_*` variable into the production browser bundle. A
`VITE_RLS_TEST_PASSWORD` would literally ship a password to every visitor. The
names are therefore plain — `RLS_TEST_A_EMAIL`, `RLS_TEST_A_PASSWORD`,
`RLS_TEST_B_EMAIL`, `RLS_TEST_B_PASSWORD` — and reach the test process via
`loadEnv` in the vitest config, which never touches the client build.

`scripts/check-bundle.mjs` (added in S1.2) is the backstop: it greps the built
bundle and fails `npm run build` on a leaked credential.

## What the test asserts

### Negative — the actual AC

Signed in as **B**, against each of A's rows (project, sprint, ticket):

| Operation | Expected |
|---|---|
| `select` by id | 0 rows |
| `update` | 0 rows affected |
| `delete` | 0 rows affected |
| `insert` a ticket into **A's project** | rejected |

That last row is not decoration. It exercises `WITH CHECK` rather than `USING` —
the asymmetry between them is the classic RLS hole, and only an insert can find it.

RLS *filters*; it does not raise. So an unauthorised update returns success with
zero rows affected, not an error. The assertions must check **affected rows**, not
absence of an error, or they pass while the database is wide open.

### Positive controls — non-negotiable

Every negative assertion is paired with a positive one: **A can** read and update
its own rows.

Without these the suite is worthless. "B sees nothing" passes just as happily when
the database is empty, the client is misconfigured, the sign-in silently failed, or
the row was never created. On a security test that false green is the worst
possible outcome — it reports safety it has not established.

### Free consequences worth asserting

- Signing in proves `handle_new_user` created a `profiles` row. (S1.2 debt.)
- A's first ticket comes back as `<KEY>-1`, proving `assign_ticket_key` fires and
  numbers atomically from `project_counters`. (S1.2 debt.)
- Updating a ticket and trying to change `key` leaves it unchanged, proving
  `freeze_ticket_key`. (S1.2 debt — added in response to the peer review.)

## Failure modes this design guards against

| Risk | Guard |
|---|---|
| Silent skip when credentials are missing → CI reports green | Skips **loudly** when unset, and **fails hard when `CI` is set**. A missing secret can never masquerade as a pass. |
| False green from an empty database | Positive controls. |
| False green from checking errors instead of row counts | Assert affected/returned row counts explicitly. |
| Run N collides with run N−1's leftovers | Project keys are unique per run. |
| Test data accumulates | Each user deletes its own projects in teardown; the cascade takes sprints, tickets, and the counter row. |

## Structure

- `src/test/rls.integration.test.ts` — `// @vitest-environment node` (network, no DOM).
- A small helper that returns a signed-in Supabase client per user, so the test
  body reads as intent rather than plumbing.
- Vitest config gains `env: loadEnv(mode, cwd, '')` so non-`VITE_` names reach the
  test process only.

## Out of scope

- CI wiring — that is S1.5 (SPRIN-13). This story leaves the suite ready for it.
- Any UI. There is none yet.
- The `profiles` read model for the assignee picker. The security review flagged
  that `handle_new_user` writes the user's **email** into `display_name` as a
  fallback, so widening the `profiles` SELECT policy — the obvious way to build
  S4.2 — would expose every user's email to every other user. **Decide that read
  model before writing S4.2**, not after. Not this story.
