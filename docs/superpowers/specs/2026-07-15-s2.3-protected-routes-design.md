# S2.3 — Protected routes and auth guard (design)

**Story (E2 Authentication):** an unauthenticated user hitting any app route is
redirected to login; an authenticated user is not redirected.

## What already exists

`RequireAuth` (built in S2.1 as the auth foundation) wraps every non-public route in
`App.tsx`. It already implements both ACs:

- `loading` → a neutral loading state (so an authenticated user is **not** bounced to
  `/login` before their session resolves);
- resolved with no session → `<Navigate to="/login" replace />`;
- resolved with a session → `<Outlet />` (the route renders).

Because every route except `/login` and `/signup` sits under `RequireAuth`, "any app
route" is covered structurally — including unknown deep paths, which match the
catch-all inside the guarded subtree.

## Scope

This story is **the guard's dedicated test coverage** — S2.1 shipped the component and
explicitly deferred these assertions here. No production behaviour change.

**Deliberately out of scope: "return to the intended destination after login."** With a
single protected route today, it has no user-visible effect; building it now is the
premature generalisation CLAUDE.md warns against. It belongs with E3, when there are
multiple protected routes to return to. The door is open (`RequireAuth` can later pass
`state={{ from: location }}`); we do not walk through it now.

## Changes

| File | Change |
|---|---|
| `src/routes/RequireAuth.test.tsx` | **New.** Mocks `useAuth` and asserts, via a small `Routes` tree: loading → neither content nor a premature `/login`; unauthenticated + `/` → `/login`; unauthenticated + a deep unknown protected path → `/login` (proves "any route"); authenticated → the route renders, no redirect, for both a known and a deep path. |

`App.test.tsx` already covers the integrated redirect/persist/logout flows; this adds
focused unit coverage of the guard's own state machine.
