# S7.3 Blocked marker and blocked-only filter — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Blocked cards surface their reason on hover, and a board toolbar checkbox filters the board to blocked-only. Closes E7.

**Architecture:** Three focused changes, no data-layer or schema touch. (1) `BlockedBadge` gains a `reason?` prop → native `title`; `TicketCard` passes `ticket.blocked_reason`. (2) `board.ts` gains a pure `selectBlockedTickets` selector, sibling to `selectActiveSprint`. (3) `BoardTab` holds local `blockedOnly` state, renders a native checkbox (active-sprint only), and applies the selector before the column split.

**Tech Stack:** React + TypeScript (strict), Vitest + Testing Library, Tailwind. Native form controls (no shadcn Checkbox exists).

/**
 * The blocked-only board filter, in one place: the tickets whose `is_blocked` flag is set.
 * Kept here beside `selectActiveSprint` rather than inlined in `BoardTab`, so "which tickets
 * are visible" stays a named, tested selector (CLAUDE.md forbids inlining domain rules in
 * components). Blocked is a flag, never a column — this narrows the set, it never moves a card.

export function selectBlockedTickets(tickets: readonly Ticket[]): Ticket[] {
  return tickets.filter((t) => t.is_blocked)
}
