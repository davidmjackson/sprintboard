# S4.6 Surface a Failed Ticket Read Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A failed ticket read shows an honest error with a working in-app Retry, instead of a confident empty backlog, an empty board, and a `0` sprint count.

**Architecture:** Give the shell's ticket read the tagged three-state shape the sprint read already has (`loading | loaded | failed`), replace `loadingTickets` with `ticketsPhase` on the context so every consumer must consider failure, and re-run both reads via a `reloadNonce` carried **in the loaded tag** so "loading" stays derived. One shared `LoadFailure` component carries the message and the Retry button.

**Tech Stack:** React 19, TypeScript strict, Vite, Tailwind, shadcn/`radix-nova` over the unified `radix-ui` package, Supabase JS, Vitest + Testing Library.

Spec: `docs/superpowers/specs/2026-07-16-s4.6-surface-failed-ticket-read-design.md`.

### Task 2: Three-state ticket read, the reload nonce, and `onRetry`

**Files:**
- Modify: `src/routes/ProjectShell.tsx`
- Modify (to compile + preserve current behaviour): `src/routes/BacklogTab.tsx`, `src/routes/SprintsTab.tsx`
- Test: `src/routes/ProjectShell.test.tsx`

**Interfaces:**
- Produces, exported from `src/routes/ProjectShell.tsx`:
```ts
export type TicketsPhase = 'loading' | 'loaded' | 'failed'
```
- On `ProjectShellContext`: **`ticketsPhase: TicketsPhase` REPLACES `loadingTickets: boolean`**, and add `onRetry: () => void`.

**Why replace rather than add:** two fields encoding one fact drift. `loadingTickets: false` would keep meaning both "loaded" and "failed" for any consumer still reading it. Replacing makes every consumer a **compile error** until it has considered failure ŌĆö that is the point of the change, not a side effect. `sprintsPhase` already exists on this context; make the two reads symmetrical.

**The shape** (mirror the existing `SprintsLoaded` in this same file):

```ts
type TicketsLoaded =
  | { projectId: string; nonce: number; phase: 'loaded'; tickets: Ticket[] }
  | { projectId: string; nonce: number; phase: 'failed' }
```

**The nonce, and why it is in the tag.** The effect keys on `[activeProjectId]`, so nothing re-runs it. Add `reloadNonce` state, put it in the effect deps, and have `onRetry` bump it. **It must also be part of the tag and the match test.** Without that, retrying from `failed` leaves the stale `failed` result still matching the current project, so the error stays on screen until the new result lands ŌĆö a Retry that appears to do nothing, which is how a user ends up hammering it. With the nonce in the tag, the stale result stops matching the instant the nonce bumps, the phase derives back to `loading`, and the user gets immediate honest feedback ŌĆö **all without a synchronous `setState` in the effect**, which the lint rule forbids.

**One nonce drives BOTH reads** (tickets and sprints). Retry means "reload this project's data". This is what closes S6.2's sticky sprint read: a failed sprint read currently persists until a page refresh, purely because nothing could re-run this effect. **Add `reloadNonce` to the sprint effect's deps and its tag too**, and derive `sprintsPhase` through the same nonce match.

Derive below the `if (!project)` guard:

```tsx
const currentTickets =
  ticketsLoaded?.projectId === project.id && ticketsLoaded.nonce === reloadNonce
    ? ticketsLoaded
    : null
const ticketsPhase: TicketsPhase = currentTickets?.phase ?? 'loading'
const tickets = currentTickets?.phase === 'loaded' ? currentTickets.tickets : []
```

**ŌÜĀ The state mutators must learn the phase.** `onTicketUpdated`, `onTicketDeleted`, and the `CreateTicketDialog` `onCreated` append all call `setLoaded` and read `prev.tickets`. Each currently guards only on `prev.projectId === project.id`. They must **also** guard `prev.phase === 'loaded'` (and preserve the nonce), or they will construct a `loaded` state out of a `failed` one, or read `.tickets` off a variant that has none. The existing `onSprintCreated` already shows this exact guard shape ŌĆö follow it.

**ŌÜĀ Update the stale doc comment.** The block above `SprintsPhase` in this file explains at length that the ticket read is deliberately *not* three-state and that fixing it is "its own story with its own debt". **This is that story.** Rewrite it: the two reads are now symmetrical. Leaving it would actively mislead the next reader. Same for the "what this does NOT fix" comment in `SprintsTab.tsx` and the long comment in `BacklogTab.tsx`'s empty state.

**In this task, keep `BacklogTab`/`SprintsTab` behaviour as it is today** ŌĆö just port them off `loadingTickets` onto `ticketsPhase` so the build is green (`ticketsPhase === 'loading'` where they said `loadingTickets`). Their error UI is Tasks 3 and 4.
