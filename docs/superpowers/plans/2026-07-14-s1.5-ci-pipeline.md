# S1.5 CI Pipeline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make a failing check physically block a merge to `main`, and make "is it green?" a single command that CI and a developer run identically.

**Architecture:** One npm script (`verify`) is the sole definition of the gate. A single GitHub Actions job named `verify` runs that exact command on every PR. A ruleset on `main` requires that check to pass and requires a PR to merge, with no bypass actors. The gate's own PR is the first thing it gates.

**Tech Stack:** GitHub Actions, GitHub rulesets, npm, Node 20, Vitest, ESLint, Prettier, Supabase.

## Global Constraints

- Branch: `feat/s1.5-ci`. Draft PR **#7** is already open against `main`.
- The ruleset on `main` is **already live** (id `18912697`) and requires a status check named exactly `verify`. That check does not exist yet, so **`main` is currently frozen** — nothing can merge until this plan lands.
- The status check name is the **job name**. It must be exactly `verify` or the ruleset never matches.
- **Never read `.env.local`.** The `SECRET FILE` guard hook blocks it, and correctly. All credential values are entered by David interactively.
- Commit messages via `git commit -m` (quoted), **never a heredoc** — the guard word-splits heredoc bodies and blocks ordinary English words.
- Node 20 (`.nvmrc`). npm 10.
- The spec is `docs/superpowers/specs/2026-07-14-s1.5-ci-pipeline-design.md`. It is authoritative; this plan implements it.

### Task 2: The workflow

**Files:**

- Create: `.github/workflows/verify.yml`

**Interfaces:**

- Consumes: the `verify` npm script from Task 1.
- Produces: a status check named `verify`, which the existing `main` ruleset requires.

- [ ] **Step 1: Write the workflow**

Create `.github/workflows/verify.yml` with exactly this content:

```yaml
name: verify

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

# This job reads code and runs checks. It writes nothing.
permissions:
  contents: read

# Superseded runs are deliberately NOT cancelled. The RLS integration suite creates
# real rows in the shared Supabase project and removes them in afterAll. Cancelling a
# run mid-suite strands a project, its sprint, its tickets and its counter row in the
# live database — permanently, on every cancelled run. A wasted CI minute is cheaper
# than leaked fixtures.
concurrency:
  group: verify-${{ github.ref }}
  cancel-in-progress: false

jobs:
  # The job name below is the exact status check the `main` ruleset requires.
  #
  # Do NOT add any of the following to this job. Each one is a way for the gate to
  # report success without having verified anything:
  #
  #   if:               a skipped job concludes as `skipped`, and GitHub accepts
  #                     `skipped` as satisfying a required check. The gate would go
  #                     green having run nothing. This is the dangerous one.
  #   strategy.matrix:  renames the check to `verify (20)`, so the required context
  #                     `verify` never reports.
  #   paths: filters:   stop the workflow firing for PRs that touch only excluded
  #                     files.
  #
  # A check that never reports fails closed (the merge is blocked), so those last two
  # are loud rather than silent — but all three disarm the gate.
  verify:
    name: verify
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc
          cache: npm

      - run: npm ci

      # The single gate. Identical to what a developer runs locally, by construction:
      # there is one definition of it, in package.json, and both sides call it.
      - name: npm run verify
        run: npm run verify
        env:
          # Public by design — these ship to every browser in dist/. See .env.example.
          VITE_SUPABASE_URL: ${{ vars.VITE_SUPABASE_URL }}
          VITE_SUPABASE_ANON_KEY: ${{ vars.VITE_SUPABASE_ANON_KEY }}
          RLS_TEST_A_EMAIL: ${{ vars.RLS_TEST_A_EMAIL }}
          RLS_TEST_B_EMAIL: ${{ vars.RLS_TEST_B_EMAIL }}
          # Actually secret.
          RLS_TEST_A_PASSWORD: ${{ secrets.RLS_TEST_A_PASSWORD }}
          RLS_TEST_B_PASSWORD: ${{ secrets.RLS_TEST_B_PASSWORD }}
```

- [ ] **Step 2: Check it is valid YAML before pushing**

Run: `npx --yes js-yaml .github/workflows/verify.yml > /dev/null && echo "valid yaml"`

Expected: `valid yaml`. A syntax error here produces no workflow run at all, and therefore no `verify` check — which blocks every merge with a confusing "Expected" status rather than a clear error.

- [ ] **Step 3: Confirm formatting passes**

Run: `npx prettier --check .github/workflows/verify.yml`

Expected: `All matched files use Prettier code style!` If it fails, run `npx prettier --write .github/workflows/verify.yml`. (`format:check` covers this file, so a badly formatted workflow would fail its own gate.)

- [ ] **Step 4: Commit**

```bash
git add .github/workflows/verify.yml
git commit -m "Run npm run verify on every pull request" -m "One job, named verify, which is the status check the main ruleset requires. It carries no if:, no matrix and no paths filter, and the file says why: each of those is a way for the job to report success without running. A skipped job satisfies a required check, which is the failure mode that would actually hurt." -m "Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>"
```

```markdown
**CI runs `npm run verify`, and that is the gate.** It is a required status check on

**Never wire CI to `npm run test:unit`.** It excludes the RLS suite and needs no
```

```markdown
## Verification

Two "green" checks have been reported on this project that were not green. Both had
the same shape: the check that ran was not the check that was claimed.

- **Verification means `npm run verify`.** Never a hand-assembled subset, never a
  proxy. `tsc --noEmit` is not `npm run build` — it passed on a branch whose build was
  red. CI runs this same command, so local and CI cannot drift.
- **Compare against `origin/*`, and fetch first.** A stale local `main` made a correct

- **Never truncate output you are using as evidence.** `git show --stat | head` hid the
  file list behind a long commit message and manufactured a false alarm.
- **A surprising result is a hypothesis, not a finding.** Before acting on or reporting
  something alarming, re-derive it a second, independent way.
```

### Task 7: AC2 — prove a failing test actually blocks the merge

**Files:**

- Create then revert: `src/test/gate-counterfactual.test.ts`

This is the deliverable of the story. AC1 is satisfied by any workflow that runs; **nothing about a green run establishes AC2.** A working gate and a purely decorative one both display a green tick. So we test the negative, exactly as the RLS suite does.

- [ ] **Step 1: Write a test that must fail**

Create `src/test/gate-counterfactual.test.ts`:

```typescript
import { describe, expect, it } from 'vitest'

/**
 * Deliberately failing. Exists for exactly one push, to prove that a red `verify`
 * check blocks the merge button rather than merely colouring it red. Reverted in
 * the next commit — see the plan for S1.5.
 */
describe('the CI gate', () => {
  it('blocks a merge when a test fails', () => {
    expect('this test must fail').toBe('and the merge must be blocked')
  })
})
```

- [ ] **Step 2: Confirm it fails locally**

Run: `npm test 2>&1 | tail -5`

Expected: `Tests  1 failed | 43 passed (44)`.

- [ ] **Step 3: Push it**

```bash
git add src/test/gate-counterfactual.test.ts
git commit -m "Add a deliberately failing test to prove the gate blocks" -m "Reverted in the next commit. A green CI run says nothing about whether CI blocks anything, so the only way to establish the second acceptance criterion is to make the check red and confirm the merge is refused." -m "Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>"
git push origin feat/s1.5-ci
```

- [ ] **Step 4: Confirm the check goes red AND the merge is refused**

Wait for the run to finish, then:

```bash
gh pr view 7 --json mergeable,mergeStateStatus -q '"mergeable="+.mergeable+"  mergeStateStatus="+.mergeStateStatus'
```

Expected: `mergeable=MERGEABLE  mergeStateStatus=BLOCKED`.

Read that carefully. **`mergeable=MERGEABLE` does not mean "you may merge"** — it means only "there are no merge conflicts". `mergeStateStatus=BLOCKED` is the field that decides. Recording the wrong one here would be a textbook instance of the mistake the new Verification section warns about.

- [ ] **Step 5: Confirm GitHub actually refuses the merge**

```bash
gh pr merge 7 --squash 2>&1 | tail -3
```

Expected: the merge is **refused**, citing the unmet required check. This is the AC. If the PR merges, the gate is decorative and the story has failed — stop and fix the ruleset.

- [ ] **Step 6: Revert, and confirm the PR becomes mergeable again**

```bash
git revert --no-edit HEAD
git push origin feat/s1.5-ci
```

Then, once CI is green again:

```bash
gh pr view 7 --json mergeable,mergeStateStatus -q '"mergeable="+.mergeable+"  mergeStateStatus="+.mergeStateStatus'
```

Expected: `mergeStateStatus=CLEAN`.

Both directions are now evidenced: red blocks, green permits. The failing commit and its revert stay in the branch history as the audit trail, and are squashed away on merge.
