# S6.3 Start sprint (enforce one active) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let a user start a `future` sprint (status → `active`), and reject starting a second while one is active with a clear message, enforced by the pre-existing partial unique index.

**Architecture:** A new intent-named data function `startSprint(id)` performs an owner-scoped `update` of `sprints.status` to `active` and maps the index's `23505` rejection to a user-correctable result. A `StartSprintButton` component owns the click/pending/error UI for one row. `SprintsTab` renders it for `future` sprints only, wired to a new `onSprintUpdated` replace-by-id reducer on `ProjectShell`.

**Tech Stack:** React + TypeScript (strict), Vite, Tailwind, shadcn/ui, Supabase JS, Vitest, Testing Library.

## Global Constraints

- **No migration. Do NOT touch the schema.** The index `sprints_one_active_per_project` already exists. Never work around it (CLAUDE.md).
- **Verify with the project's scripts, never a proxy.** `npm run test:unit` for the fast loop; **`npm run build`** to typecheck (NOT `npx tsc --noEmit` — it checks zero files here). Do not run the full `npm test` in a subagent (it needs live DB secrets); the orchestrator runs `npm run verify`.
- **Never inline the four board-column names** anywhere (not relevant to this story, but standing).
- **Status/type/project_type stay `text` + `check`.** Never introduce a Postgres enum.
- **Import order:** React, then third-party, then `@/…` local, matching neighbouring files.
- **Sprint status labels come from `SPRINT_STATUS_LABELS` in `domain.ts`**, never inlined.
- Adding a **required** field to `ProjectShellContext` breaks every harness that builds the context. The sites are `src/routes/BoardTab.test.tsx` and `src/routes/SprintsTab.test.tsx` — both must gain `onSprintUpdated: vi.fn()`.

```ts
import type { Sprint, SprintCreateInsert, SprintStatusUpdate } from './domain'
```

```ts
/**
 * Start a sprint: flip its status to `active`. The one-active-per-project rule is enforced
 * by the `sprints_one_active_per_project` partial unique index, NOT by this function — we
 * attempt the update and let the database reject a second active sprint. We never deactivate
 * another sprint to make room: that would work around the index (CLAUDE.md forbids it) and
 * silently end a running sprint.
 *
 * Unlike `createSprint`, this has a user-correctable failure. A `23505` (unique_violation)
 * is the index rejecting a second active sprint — the user can finish the current one and
 * retry — so it gets its own tag and a clear message at the UI. Everything else (an RLS
 * zero-row match on a cross-tenant or missing id, a network error) is not user-correctable
 * and collapses to `'unknown'`. RLS (`sprints_owner`) scopes the write through the owned
 * project, exactly as in the browser.
 */
export type StartSprintResult =
  | { ok: true; sprint: Sprint }
  | { ok: false; error: 'already_active' }
  | { ok: false; error: 'unknown' }

export async function startSprint(id: string): Promise<StartSprintResult> {
  const { data, error } = await supabase
    .from('sprints')
    .update({ status: 'active' } satisfies SprintStatusUpdate)
    .eq('id', id)
    .select()
    .single()

  if (error) {
    if (error.code === '23505') return { ok: false, error: 'already_active' }
    return { ok: false, error: 'unknown' }
  }
  return { ok: true, sprint: data as Sprint }
}
```

- [ ] **Step 3: Implement the component** — `src/routes/StartSprintButton.tsx`:

```tsx
import { useState } from 'react'

import { startSprint } from '@/lib/sprints'
import type { Sprint } from '@/lib/domain'
import { Button } from '@/components/ui/button'

/**
 * Starts one sprint. Owns the async call, its pending state, and the error message for a
 * single row so `SprintsTab` stays a pure view. The `already_active` case (the partial
 * unique index rejecting a second active sprint) is user-correctable, so it gets a clear,
 * specific message; anything else is the generic retry copy, matching `CreateSprintDialog`.
 *
 * `setPending(false)` runs BEFORE `onStarted`: a successful start flips the sprint out of
 * `future`, so `SprintsTab` stops rendering this button and it unmounts — no state is set
 * on it afterwards.
 */
export function StartSprintButton({
  sprint,
  onStarted,
}: {
  sprint: Sprint
  onStarted: (sprint: Sprint) => void
}) {
  const [pending, setPending] = useState(false)
  const [error, setError] = useState<string | null>(null)

  async function handleStart() {
    setPending(true)
    setError(null)
    const result = await startSprint(sprint.id)
    setPending(false)
    if (result.ok) {
      onStarted(result.sprint)
      return
    }
    setError(
      result.error === 'already_active'
        ? 'This project already has an active sprint. Complete it before starting another.'
        : 'Something went wrong. Please try again.',
    )
  }

  return (
    <div className="flex shrink-0 flex-col items-end gap-1">
      <Button size="sm" variant="outline" onClick={handleStart} disabled={pending}>
        {pending ? 'Starting…' : 'Start'}
      </Button>
      {error ? (
        <p role="alert" className="text-destructive max-w-[16rem] text-right text-xs">
          {error}
        </p>
      ) : null}
    </div>
  )
}
```
